# JRN — NONOGRAM mdtpw port + resource reconstruction

**Date:** 2026-05-29
**Branch:** `main`
**Related:** `wrk_docs/2026.05.29 - HLD - NONOGRAM 16-bit to 32-bit mdtpw port.md`

## What was done

Rebuilt the 1993-97 16-bit Nonogram (Borland TPW 1.5 OWL) as a native 32-bit
PE with the mdtpw toolchain, mirroring the `stopbus` port.

### 1. Survey
- Copied `NONOGRAM.PAS` and the mdtpw `runtime_pas` units into `_survey/` for
  read-only inspection.
- First survey-compile surfaced a runtime gap rather than an app bug:
  `survey_log.txt` — `WinTypes.pas:531: unknown type TDEVMODE`. Noted as an
  mdtpw runtime_pas issue, not a nonogram source issue; did not block the port.

### 2. Build bring-up
- `build_log.txt`: mdtpw rejected `FileVar: FILE OF CHAR;`
  ("expected `;` after var declaration") inside `FindBytes`/`CheckLicence`.
  The construct is Turbo Pascal typed-file I/O used *only* by the licence
  reader. Because the licence gate was going to be removed anyway, stripping
  it both achieved the product goal and sidestepped the parser limitation —
  one change, two wins.

### 3. Source edits (`NONOGRAM.PAS`, +8 / -233)
- Removed `FindBytes` + `CheckLicence`; start the app directly via
  `NonogramApp.Init/Run/Done` (was gated behind `IF CheckLicence`).
- `MaxXAcross`/`MaxYAcross` 64 → 100, `MaxNoElements` 15 → 40 to fit the
  larger bundled puzzles (MILLENNI 75×55, PUMPKIN run lines).
- Copyright glyph byte tidy-up in the About box.
- Left the canonical source as `NONOGRAM.PAS`; the loose `nonogram*.pas`
  working copies are scratch and are now gitignored.

### 4. Resource reconstruction
- The 16-bit `NONOGRAM.RES` (Win16 record format, `FF 02 …`) is rejected by
  `lld-link`. Reconstructed a Win32 resource with mdtpw's `nonores`:
  `generated/nonogram.rc` + 5 toolbar/button `.bmp` and 4 `.ico` assets.
- Tracked `generated/` as a build *input*; `mdtpw.toml` references it and the
  pipeline auto-compiles it via the pinned SDK `rc.exe`.
- **Decision:** reverted the working-tree `NONOGRAM.RES` (someone had replaced
  it with rc.exe's Win32 output) back to the tracked 16-bit original — the
  historical artifact stays tracked; the derived Win32 `.res` is rebuilt from
  `generated/nonogram.rc` and never committed (stopbus precedent).

### 5. Build config + hygiene
- Added `mdtpw.toml` (`src=NONOGRAM.PAS`, `output=nonogram_mdtpw.exe`,
  `subsystem=windows`, `resources=["generated/nonogram.rc"]`). `output` is
  suffixed so the build never clobbers the tracked 16-bit `NONOGRAM.EXE`.
- Added `.gitignore`: build output (`*.ll`, `*.obj`, `nonogram_mdtpw.exe`),
  scratch source copies, `NONOGRAM.RES.orig16`, `NONOGRAM.INI`, logs, and the
  `_diag/` `_survey/` `shots/` scratch dirs.

## Open items / risks
- mdtpw `runtime_pas` `TDEVMODE` gap (from the survey) is unaddressed here —
  belongs to the mdtpw project, not nonogram.
- Full puzzle-library solve sweep still in progress (see solver-verification
