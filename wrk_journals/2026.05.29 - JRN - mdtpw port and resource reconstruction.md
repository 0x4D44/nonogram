# JRN — NONOGRAM mdtpw port + resource reconstruction

**Date:** 2026-05-29
**Branch:** `main`
**Related:** `wrk_docs/2026.05.29 - HLD - NONOGRAM 16-bit to 32-bit mdtpw port.md`

## What was done

Rebuilt the 1993-97 16-bit Nonogram (Borland TPW 1.5 OWL) as a native 32-bit
PE with the mdtpw toolchain, mirroring the `stopbus` port.

### 1. Survey
- Copied `NONOGRAM.PAS` and the mdtpw `runtime_pas` units into `_survey/` for
  read-only inspection.
- First survey-compile surfaced a runtime gap rather than an app bug:
  `survey_log.txt` — `WinTypes.pas:531: unknown type TDEVMODE`. Noted as an
  mdtpw runtime_pas issue, not a nonogram source issue; did not block the port.

### 2. Build bring-up
- `build_log.txt`: mdtpw rejected `FileVar: FILE OF CHAR;`
  ("expected `;` after var declaration") inside `FindBytes`/`CheckLicence`.
  The construct is Turbo Pascal typed-file I/O used *only* by the licence
  reader. Because the licence gate was going to be removed anyway, stripping
  it both achieved the product goal and sidestepped the parser limitation —
  one change, two wins.

### 3. Source edits (`NONOGRAM.PAS`, +8 / -233)
- Removed `FindBytes` + `CheckLicence`; start the app directly via
  `NonogramApp.Init/Run/Done` (was gated behind `IF CheckLicence`).
- `MaxXAcross`/`MaxYAcross` 64 → 100, `MaxNoElements` 15 → 40 to fit the
  larger bundled puzzles (MILLENNI 75×55, PUMPKIN run lines).
- Copyright glyph byte tidy-up in the About box.
- Left the canonical source as `NONOGRAM.PAS`; the loose `nonogram*.pas`
  working copies are scratch and are now gitignored.

### 4. Resource reconstruction
- The 16-bit `NONOGRAM.RES` (Win16 record format, `FF 02 …`) is rejected by
  `lld-link`. Reconstructed a Win32 resource with mdtpw's `nonores`:
  `generated/nonogram.rc` + 5 toolbar/button `.bmp` and 4 `.ico` assets.
- Tracked `generated/` as a build *input*; `mdtpw.toml` references it and the
  pipeline auto-compiles it via the pinned SDK `rc.exe`.
- **Decision:** reverted the working-tree `NONOGRAM.RES` (someone had replaced
  it with rc.exe's Win32 output) back to the tracked 16-bit original — the
  historical artifact stays tracked; the derived Win32 `.res` is rebuilt from
  `generated/nonogram.rc` and never committed (stopbus precedent).

### 5. Build config + hygiene
- Added `mdtpw.toml` (`src=NONOGRAM.PAS`, `output=nonogram_mdtpw.exe`,
  `subsystem=windows`, `resources=["generated/nonogram.rc"]`). `output` is
  suffixed so the build never clobbers the tracked 16-bit `NONOGRAM.EXE`.
- Added `.gitignore`: build output (`*.ll`, `*.obj`, `nonogram_mdtpw.exe`),
  scratch source copies, `NONOGRAM.RES.orig16`, `NONOGRAM.INI`, logs, and the
  `_diag/` `_survey/` `shots/` scratch dirs.

## Build bring-up status (post-mdtpw.toml)

Running `mdtpw` in the repo now compiles and reaches the link stage.

1. **Resource binding (fixed, nonogram-side).** Disabled the
   `{$R NONOGRAM.RES}` directive (NONOGRAM.PAS:58). mdtpw honours `{$R}` and
   was handing the 16-bit (Win16) NONOGRAM.RES straight to lld-link
   ("unknown file type"). Resources are now bound solely via `mdtpw.toml`
   (`generated/nonogram.rc` → rc.exe → Win32 `.res`). Resource error resolved.

2. **Remaining lld-link errors are the in-progress mdtpw bring-up, not
   nonogram source bugs:**
   - `__tpw_file_read_chararr`, `__tpw_val_int`, `__tpw_strpas` — RTL
     intrinsics the compiler emits for typed-file char-array `Read` / `Val` /
     `StrPas`. Implemented on the mdtpw `worktree-coverage-98` branch (code
     comments cite "Nonogram bring-up HLD §3.1" + nonogram.pas line numbers)
     but **not** in mdtpw `main`, so the installed `/c/apps/mdtpw` is stale.
     Fix: rebuild + install mdtpw once that branch lands. Concurrent work —
     not touched.
   - `_FileExists@4`, `_ConvertDate@8` — the repo's `MDGENTPU.PAS` imports
     these from the 16-bit MDGen DLL (`EXTERNAL 'MDGen' INDEX 2/3`). mdtpw
     ships a native `runtime_pas/MDGentpu.pas`, but mdtpw resolves units
     source-dir-first so the repo copy shadows it. Resolution is a design
     decision (use the native unit without dropping the historical
     `MDGENTPU.PAS`) — deferred to discuss with Arthur.

## Open items / risks
- 32-bit build is **not green yet**: blocked on the two mdtpw-side items above
  (stale installed mdtpw + MDGentpu unit shadowing). nonogram source compiles.
- mdtpw `runtime_pas` `TDEVMODE` gap (from the survey) is unaddressed here —
  belongs to the mdtpw project, not nonogram.
- Full puzzle-library solve sweep still in progress (see solver-verification
  journal); MILLENNI does not solve (provably unsatisfiable + solver hang).

## Update 2026-05-30 — 32-bit build is GREEN

Both blockers from the bring-up status above are resolved; `mdtpw` now links
`nonogram_mdtpw.exe` cleanly (exit 0, ~320 KB PE32).

- **`__tpw_*` RTL intrinsics:** already fixed — the installed `C:\apps\mdtpw.exe`
  was rebuilt since the 05-29 note, so those symbols resolve. No longer in the error.
- **`_FileExists@4` / `_ConvertDate@8` (the deferred design item):** root cause was
  *not* unit shadowing as first suspected. mdtpw's `prefer_runtime` carve-out
  (`mdtpw/src/lib.rs:344`) already lists `MDGentpu` alongside `PXEngine`/`Ctl3d`,
  so the native `runtime_pas/MDGentpu.pas` *should* override the local 16-bit
  `MDGENTPU.PAS`. But `MDGentpu.pas` was missing from the **embedded** unit table
  `RUNTIME_PAS` in `mdtpw/src/embedded.rs` (a deployed binary with no reachable
  `runtime_pas/` sibling resolves carve-out units only via the embedded copy —
  `find_runtime_pas` Pass 3). `Ctl3d`/`PXEngine` were embedded; `MDGentpu` was the
  one omission — an incomplete change. So the carve-out silently fell back to the
  local import wrapper → undefined `MDGen` DLL imports.
  - **Fix (mdtpw-side, one line):** added `("MDGentpu.pas", include_bytes!(…))` to
    `RUNTIME_PAS` in `mdtpw/src/embedded.rs`; rebuilt + reinstalled `C:\apps\mdtpw.exe`.
  - **Verified:** regenerated `nonogram_mdtpw.ll` now emits `define @tpw.fileexists`
    / `define @tpw.convertdate` (bodies, not external `declare`s) plus the
    `ConstructDate` helper that exists *only* in the native unit — proof the native
    unit compiled in.
  - **No nonogram-repo change:** the historical 16-bit `MDGENTPU.PAS` stays tracked
    exactly as intended; the carve-out prefers the runtime unit over it.
  - **Residual:** the `embedded.rs` one-liner is **uncommitted** in the mdtpw repo
    (sitting atop unrelated in-flight WIP) — owner to commit (suggest patch bump
    0.12.0 → 0.12.1). Also fixes any other deployed-binary `MDGentpu` consumer (stopbus).
