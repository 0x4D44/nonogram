# JRN - Window and file dialog fixes

**Date:** 2026-05-30
**Repo:** `C:\language\nonogram`

## Initial investigation

Arthur reported three GUI problems in the mdtpw-built Nonogram app: the title
feels slow to appear, the MDI surface looks wrong, and File|Open hangs.

There is no repo-local `AGENTS.md`; I am following the instructions supplied in
the prompt. The worktree was already dirty on entry (`NONOGRAM.EXE`,
`mdtpw.toml`, an existing journal, and untracked `WIN16/`), so I am treating
those as someone else's in-flight work unless I have to touch them.

The generated IR shows the translated OWL command dispatch invokes the frame's
`CM_LOAD` handler before the MDI frame's default command forwarding can send the
command to the active child. That makes `TMainWindow.CMLoad` post `CM_LOAD` back
to itself after creating a child, which explains the apparent hang: it can keep
creating/activating children instead of reaching `TDisplay.CMLoad` and opening
the common dialog.

The capture script also reproduced the visible layout problem: the toolbar and
status bar window rectangles are far offscreen after startup. The only app-side
negative moves are the `MoveWindow(..., -1, -1, ...)` calls in
`TMainWindow.HandleSizing`; in the translated Win32 ABI those signed 16-bit
arguments are not safe. My frustration level is moderate here because this is a
toolchain-shaped compatibility wart, but the app-side workaround is small and
defensible.

## Fix and verification

Changed `TMainWindow.CMLoad` so File|Open creates one new display child and then
calls `DisplayHan^.CMLoad(Msg)` directly. This avoids the translated command
dispatcher re-entering the frame's `CM_LOAD` handler. A temporary probe posted
`CM_LOAD` to the frame and confirmed a real `Open` common dialog appears.

Changed `TMainWindow.HandleSizing` to move toolbar/status windows to
non-negative coordinates. Rebuilt with `mdtpw` successfully. A post-fix
`capture-gui.ps1` run showed toolbar, toolbuttons, status bar, MDI client, and
the display window all in sane visible rectangles. The screenshot also looks
substantially better: the default child is now visually integrated with the MDI
client instead of showing a stray internal titlebar over an empty top band.

The existing one-puzzle smoke script had a stale dialog edit-control ID (`1148`);
the current dialog uses `1152`, matching `solve-all.ps1`. I updated
`solve-puzzle.ps1` to accept both IDs and reran it against `MARYMARY.NGM` using
the rebuilt `nonogram.exe`. It loaded and solved successfully, producing
`marymary_post_fix_loaded.png` and `marymary_post_fix_solved.png`. Emotional
temperature is lower now: this was a clean pair of app-side compatibility fixes,
not a broad MDI rewrite.

## Follow-up: MDI and dialog rendering

Arthur found more GUI oddities: the information dialog had a dark bar over the data title, Preferences had clipped-looking group box artefacts, the Window menu commands did not act on MDI children, File|Open left a new empty child after Cancel, and the About box showed a `?` instead of the copyright mark.

Current hypothesis held up: several reconstructed resource IDs/styles were wrong, and the frame was relying on translated OWL defaults that are too thin for the Window menu commands. I changed the Window menu resource IDs to the OWL MDI command IDs, added explicit frame handlers for tile/cascade/arrange/close-all, moved File|Open so the dialog runs before creating a child, removed the bogus black static rectangle from the information dialog, adjusted the Preferences group-box rectangle, and changed the copyright text to ASCII `(c)`. Emotional temperature: mildly annoyed at the resource reconstruction edge cases, but this is still bounded and testable.

## Final validation notes

The Window menu needed one extra pass. The Win16 code packed two HMENU values
into `MAKELONG` for `WM_MDISETMENU`; in the Win32 port that corrupts the menu
handle path, so the frame now uses a direct USER32 `SetMenu` call. The MDI child
menu entries also carry full 32-bit IDs in each child window's `GWL_ID`, while
the OWL `WM_COMMAND` repack exposes only the low word. I added a frame-level
`WM_COMMAND` override for 3841-3849 that maps the low-word menu number back to a
real child HWND by scanning the MDICLIENT children and matching `LOWORD(GWL_ID)`,
then sends `WM_MDIACTIVATE` with a full-width HWND.

The final probe verified: File|Open Cancel leaves the child count unchanged;
Window menu IDs 3841-3844 switch between the expected children, including
`3842 -> MARYMARY.NGM`; Tile and Cascade move child windows; the Info dialog no
longer has the black title overpaint; Preferences no longer has the clipped
group-box strips; About displays `Copyright (c) MD Soft 1993-1997`. I also reran
`solve-puzzle.ps1` against `MARYMARY.NGM` and the puzzle loaded and solved.

After the visual pass I removed one more bogus reconstructed InfoBox control: a
2-pixel-high hidden `Cancel` button that left a tiny mark in the Data title
area. Rebuilt again and reran the `MARYMARY.NGM` solve smoke successfully.

Residual concern: `generated/nonogram.rc` is still a patched generated artifact.
If `cargo run --bin nonores` or the external mdtpw resource generator is rerun,
these resource fixes may need to be moved into the generator. Emotional
temperature: calmer after the GWL_ID discovery; that made the MDI selection
fix feel grounded rather than guessed.
