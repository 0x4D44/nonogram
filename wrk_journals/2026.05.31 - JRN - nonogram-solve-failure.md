# 2026-05-31 - Nonogram solve failure investigation

## Initial notes
- Arthur reports that opening many `.NGM` files shows "Data cannot be solved: use the Information menu option for more help".
- Example file in the dialog: `C:\language\nonogram\ENGLISH.NGM`.
- First goal: reproduce and identify whether this is bad puzzle data, parser damage, solver logic, or a UI/reporting issue.

## Emotional temperature
- Curious and slightly wary: this looks like legacy Pascal/Win16-era code where a tiny indexing or parsing assumption can have broad effects.
- Main frustration risk is confusing generated/transpiled artifacts with the real source; I will avoid changing generated outputs until the source path is clear.

## Findings
- The dialog is raised from `TDisplay.FileLoadImplement` after `ParseData`, specifically when `Sum1 <> Sum2` or either sum is zero.
- Static parsing of all workspace `.NGM` files found 55 internally consistent files and 3 inconsistent files.
- `ENGLISH.NGM`: 35x35, 70 clue lines, first 35 clue lines sum to 651, last 35 clue lines sum to 795. The file has the expected number of lines, but the clue totals cannot describe one grid.
- `XMAS.NGM`: 60x50, 110 clue lines, first 60 clue lines sum to 1110, last 50 clue lines sum to 1505. Each individual clue line fits its axis, but the total filled cells disagree.
- `GREETING.NGM`: currently worktree-modified and badly damaged: first 60 clue lines are blank, one clue line contains `535`, and totals are impossible. The tracked version is also not a valid solved puzzle input; it has blank leading clue lines and excess trailing lines.
- GUI automation reproduced the report with `ENGLISH.NGM` using `nonogram.exe`, and a control run with `MARYMARY.NGM` loaded and solved without the parse warning.

## Current hypothesis
- The message is correct for these files. The app is not rejecting "lots" of valid bundled puzzles; it is rejecting a small number of invalid/root-level `.NGM` files that are outside `NGMS.ZIP` or have been locally generated/edited.

## Emotional temperature update
- More settled now: the signal is clean. Mild irritation remains around `GREETING.NGM` because it is both tracked-corrupt and currently more damaged, but it is unrelated to the reported `ENGLISH.NGM` dialog.
