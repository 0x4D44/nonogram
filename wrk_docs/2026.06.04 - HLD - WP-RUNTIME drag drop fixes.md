## 0. Build mechanism — how the runtime is consumed/rebuilt (ESSENTIAL)

This is load-bearing for the verify plan, so it is stated up front.

The Pascal runtime units in `C:\language\mdtpw\runtime_pas\*.pas` (incl.
`WObjects.pas`, `ShellAPI.pas`) are **compiled from source on every app
build** — they are *not* precompiled into a `.lib`. The native `mdtpw_rt`
staticlib (`C:\language\mdtpw\mdtpw_rt\src\lib.rs`, the `#![no_std]` Win32
floor) contains only the System/WinCrt runtime, no OWL/Shell logic
(`mdtpw_rt\Cargo.toml`). So no `cargo`-level relink of the floor is needed for
a `.pas` edit.

**BUT** the `.pas` sources are also `include_bytes!`-baked into `mdtpw.exe`
at *its* compile time (`C:\language\mdtpw\src\embedded.rs:12` `RUNTIME_PAS`,
each `include_bytes!(".../runtime_pas/<unit>.pas")`). At app-build time the
compiler resolves each unit via `find_runtime_pas`
(`C:\language\mdtpw\src\lib.rs:219`) in this priority order:

1. **Pass 1** — walk up from the *source* dir (`C:\language\nonogram`) looking
   for a `runtime_pas\<unit>.pas`. There is none on that path → miss.
2. **Pass 2** — walk up from the *`mdtpw.exe` binary's* dir.
3. **Pass 3** — the extracted embedded snapshot under
   `%LOCALAPPDATA%\mdtpw\<fnv-hash>\runtime_pas\` (`embedded.rs:176`
   `ensure_extracted`; `lib.rs:247`).

**Which binary builds Nonogram?** `mdtpw` on PATH resolves to
`C:\apps\mdtpw.exe` (`where mdtpw`), the *installed* binary — **not** the
in-tree `C:\language\mdtpw\target\release\mdtpw.exe`. `C:\apps\` has **no
`runtime_pas\` sibling**, so for the installed binary passes 1 and 2 both
miss and the compiler serves the **Pass-3 embedded snapshot** frozen into
`C:\apps\mdtpw.exe` when it was last built.

**Consequence (the rule for this WP):** editing
`C:\language\mdtpw\runtime_pas\*.pas` has **NO effect on a Nonogram build run
via `C:\apps\mdtpw.exe`** until the snapshot is refreshed. To make the edit
take effect there are two supported routes; this WP uses route (A):

- **(A) Rebuild + reinstall the compiler.** `cargo build --release` in
  `C:\language\mdtpw` re-runs `build.rs` (floor staticlibs) and re-embeds the
  edited `.pas` via `include_bytes!`, producing a fresh
  `target\release\mdtpw.exe` whose embedded FNV hash changes (so it extracts
  to a *new* `%LOCALAPPDATA%\mdtpw\<hash>\` — old caches untouched, per
  `dist\README.md` §"Cache directory"). Then copy that exe over
  `C:\apps\mdtpw.exe`. This is the route the dist README documents
  ("copy it anywhere on the target host's PATH").
- **(B) Filesystem override (dev only).** Drop the edited `runtime_pas\`
  beside the exe or above the source so Pass 1/2 wins — *not* used here
  because the PATH binary lives in `C:\apps` and we want the verified,
  reproducible installed-binary path.

**So: yes, a `cargo build` of mdtpw IS required for the runtime change, and it
must be sequenced BEFORE the app build, with the rebuilt exe installed to
`C:\apps`.** See §6 for the exact ordering. `mdtpw_rt` does *not* need its own
`cargo build -p` step (no floor symbols change).

## 1. Problem statement

Dragging a `.NGM` file onto the Nonogram main window does nothing.
`TMainWindow.SetupWindow` registers for drops via `DragAcceptFiles(HWindow,
TRUE)` (`NONOGRAM.PAS:4567`) and `TMainWindow.WMDropFiles`
(`NONOGRAM.PAS:4749`) is DDVT-bound to `WM_DROPFILES`
(`NONOGRAM.PAS:416-417`, `virtual WM_FIRST + WM_DROPFILES`), but the handler
never opens the dropped file. Two independent, compounding defects, one in
each repo, together break the feature. Both must be fixed and verified
together.

## 3. Findings

### Finding M (BLOCKER) — 32-bit HDROP truncated to 16 bits in delivery

**(1) Defect + exact file:line (both repos).**
- App: `C:\language\nonogram\NONOGRAM.PAS:4755`, `:4762`, `:4767` read the
  `HDROP` from `Msg.WParam`.
- Runtime: `C:\language\mdtpw\runtime_pas\WObjects.pas:302-309` —
  `TMessage.WParam` is a 16-bit `Word` at offset 6 (case-0 variant of the
  Win16-shaped record). `WM_DROPFILES` ($0233, `WinTypes.pas:92`) is **not**
  special-cased in `RepackWin32` (`WObjects.pas:1795-1943`), so it falls into
  the **default arm** at `WObjects.pas:1940`: `Msg.WParam := LoWord(AWParam)`
  (`LoWord` = `L and $FFFF`, `WinProcs.pas:938-941`).

**(2) Root cause.** The real Win32 `WM_DROPFILES` carries the `HDROP` in the
full 32-bit `wParam`. `RepackWin32`'s default arm narrows `wParam` to the
Win16 16-bit `Word`, discarding the high 16 bits of the handle. The handler
then hands a corrupted handle to `DragQueryFile`/`DragFinish`, which fail.
The fix from STOPBUS's `WMEraseBkgnd` (same class of bug — a 32-bit HDC could
not ride the 16-bit `WParam`) is the precedent: the runtime already latches
the **untruncated** Win32 args.

**Key fact (verified):** `OwlDispatchCore` (`WObjects.pas:1996`) sets the
`DefWParam`/`DefLParam` latch (`WObjects.pas:1724-1725`, module-global
`TWinParam` = `LongInt` on win32 / `Pointer` on win64) **unconditionally,
before dispatch** (`WObjects.pas:2023-2024` `DefWParam := AWParam;
DefLParam := ALParam;`) to the **raw, full-width** Win32 args. So while
`WMDropFiles` runs, `DefWParam` already holds the untruncated 32-bit `HDROP`.
The runtime's own `WMEraseBkgnd` reads the 32-bit HDC straight from
`DefWParam` for exactly this reason (`WObjects.pas:2784,2787`).

There is one published accessor for the latch: `function CurrentLParam:
TWinParam` (`WObjects.pas:1092` decl / `:1729` body), already consumed by
`StdDlgs.pas:82`. There is **no** `CurrentWParam` yet.

**(3) Fix — runtime + app sides.**

*Runtime side (add the symmetric accessor; smallest backward-compatible
change):* expose the latched 32-bit `wParam` via a new `CurrentWParam`,
mirroring the existing `CurrentLParam` exactly.

`WObjects.pas` interface — *before* (`:1092`):
```pascal
function CurrentLParam: TWinParam;
```
*after:*
```pascal
function CurrentLParam: TWinParam;
function CurrentWParam: TWinParam;
```

`WObjects.pas` implementation — *before* (`:1729-1732`):
```pascal
function CurrentLParam: TWinParam;
begin
  CurrentLParam := DefLParam;
end;
```
*after:* add immediately below it:
```pascal
function CurrentWParam: TWinParam;
begin
  CurrentWParam := DefWParam;
end;
```
(Insertion points: one interface line after `WObjects.pas:1092`; one 4-line
body after `WObjects.pas:1732`. No change to `RepackWin32`, no new
`WM_DROPFILES` case, no `TMessage` layout change.)

*App side:* read the full-width `HDROP` from `CurrentWParam` instead of the
truncated `Msg.WParam`. `NONOGRAM.PAS:4749-4768` — *before:*
```pascal
 NumFiles := DragQueryFile(Msg.WParam, $FFFF, NIL, 0);
 ...
   DragQueryFile(Msg.WParam, I, FileName, Pred(Sizeof(FileName)));
 ...
 DragFinish(Msg.wParam);
```
*after* (capture the latched handle once, then use it; combined with
Finding N below):
```pascal
PROCEDURE TMainWindow.WMDropFiles(VAR Msg: TMessage);
VAR
 Drop: HDROP;
 NumFiles, I: LongInt;
 FileName: ARRAY [0..130] OF CHAR;
BEGIN
 Drop := HDROP(CurrentWParam);            (* full 32-bit HDROP, not truncated *)
 (* Firstly find out how many files were dropped *)
 NumFiles := DragQueryFile(Drop, $FFFFFFFF, NIL, 0);   (* UINT(-1) count query *)

 IF (IsIconic(HWindow)) THEN OpenIcon(HWindow);

 (* Now we can retrieve the dropped files and open them *)
 IF (NumFiles > 0) THEN
   FOR I := 0 TO (NumFiles-1) DO
   BEGIN
     DragQueryFile(Drop, I, FileName, Pred(Sizeof(FileName)));
     FileOpener(FileName);
   END;

 (* Cleanup - tell Windows that we're done with the "drop info" *)
 DragFinish(Drop);
END;
```
`HDROP` is already in scope via `USES ... ShellAPI` (`NONOGRAM.PAS:65`);
`CurrentWParam` via `USES ... WObjects`. `IsIconic`/`OpenIcon`/`FileOpener`
(`NONOGRAM.PAS:5220`) are unchanged.

**(4) Alternatives + why rejected.**
- *(a) Add a `WM_DROPFILES` special case to `RepackWin32` that stows the
  32-bit HDROP in `Msg.LParam`.* `LParam` is genuinely free for
  `WM_DROPFILES` (Win32 lParam is unused), and it is 32-bit, so it could
  carry the handle. Rejected as the primary because (i) it changes the proven
  S3 repack table and adds a per-message arm where the `CurrentWParam`
  accessor needs none; (ii) it diverges from the STOPBUS `WMEraseBkgnd`
  precedent (which reads the latch, not a repacked field); (iii) the app
  would then read `Msg.LParam`, which is *less* self-documenting than
  "current wParam". Kept as a fallback if the reviewer prefers zero new
  runtime exports (it is also backward-compatible — no existing handler reads
  `WM_DROPFILES` `Msg.LParam`).
- *(b) Read `DefWParam` directly in the app.* Impossible — `DefWParam` is a
  module-global in `WObjects`, not exported; only `CurrentLParam`/(new)
  `CurrentWParam` cross the unit boundary.
- *(c) Widen `TMessage.WParam` to 32-bit.* Rejected outright — breaks the
  proven Win16 record geometry (`SizeOf=16`, field offsets at
  `WObjects.pas:298-308`) that the entire corpus depends on.

### Finding N (issue) — `DragQueryFile` signature + WORD underflow

**(1) Defect + exact file:line (both repos).**
- Runtime: `C:\language\mdtpw\runtime_pas\ShellAPI.pas:41-43` declares
  `function DragQueryFile(HDrop: HDROP; iFile: Word; ...)` bound to
  `'SHELL32' name 'DragQueryFileA'`.
- App: `NONOGRAM.PAS:4755` passes `$FFFF` as the count sentinel; `:4751`
  declares `NumFiles, I: WORD`; `:4760` loops `FOR I:=0 TO (NumFiles-1)`.

**(2) Root cause.** Win32 `DragQueryFileA`'s count query requires
`iFile = 0xFFFFFFFF` (`UINT(-1)`). A `Word` parameter cannot hold it: the app
already truncates to `$FFFF`, and even if it passed `$FFFFFFFF` the `Word`
param would truncate it to `$FFFF` = "give me file index 65535" → API returns
0/garbage. Separately, with `NumFiles: WORD`, when the (correct) count is 0
the loop bound `NumFiles-1` underflows to `$FFFF` (65535) and the `FOR` runs
65535 garbage iterations.

**(3) Fix — runtime + app sides.**

*Runtime side* — widen the count/index parameter to 32-bit so it can carry
`UINT(-1)`. `ShellAPI.pas:41` *before:*
```pascal
function DragQueryFile(HDrop: HDROP; iFile: Word; lpszFile: PChar;
  cch: Integer): Integer;
  external 'SHELL32' name 'DragQueryFileA';
```
*after:*
```pascal
function DragQueryFile(HDrop: HDROP; iFile: LongInt; lpszFile: PChar;
  cch: Integer): Integer;
  external 'SHELL32' name 'DragQueryFileA';
```
(`LongInt` chosen over `Cardinal` to match the surrounding declarations'
signed-`Integer`/`LongInt` house style; the bit pattern `$FFFFFFFF` is
identical either way at the ABI. `iFile` is the only edit — the rest of the
declaration, including the `external '...' name 'DragQueryFileA'` binding, is
byte-for-byte unchanged.)

*App side* — pass the proper sentinel and guard the loop. Already folded into
the Finding-M sketch above: `$FFFF` → `$FFFFFFFF`; `NumFiles, I: WORD` →
`LongInt`; add the `IF (NumFiles > 0) THEN` guard around the `FOR`.
(Widening `NumFiles` to a signed 32-bit `LongInt` *also* fixes the underflow
intrinsically — `0-1 = -1`, and the `FOR 0 TO -1` simply does not execute —
so the explicit `IF` is belt-and-braces against a garbage count from a
hostile/odd drop. Open question §7 asks whether the reviewer wants the guard
*and* the widening, or just one.)

**(4) Alternatives + why rejected.**
- *Keep `iFile: Word`, special-case the count query.* No legal `Word`
  value is `UINT(-1)`; you cannot query the count at all. Rejected.
- *Cast `$FFFFFFFF` at the call site without widening the param.* The `Word`
  formal truncates it back to `$FFFF` regardless of the call-site cast.
  Rejected.

## 4. ABI / binding confirmation (verified)

Widening `iFile` does **not** change the external binding. mdtpw emits
external imports by their `name '...'` string under `x86_stdcallcc` for win32
(`emit.rs:572-615`); the import slot is keyed on `DragQueryFileA`, unchanged.
At the IR level the param type is emitted via `tenv.llvm(p.ty)`
(`emit.rs:605`): a `Word` declares as `i16`, a `LongInt` as `i32`. Real
`DragQueryFileA` takes a 32-bit `UINT`, so the current `Word`/`i16`
declaration is *already* an ABI under-width; widening to `i32` makes the
declaration **match** the real Win32 ABI — a strict correctness improvement,
not new churn. `DragFinish`/`DragAcceptFiles`/`DragQueryPoint`/`ShellExecute`
signatures are untouched.

## 5. Blast radius — who else could a `RepackWin32`/`ShellAPI` change affect?

`WObjects.pas` and `ShellAPI.pas` are shared by the whole TPW corpus, so this
matters.

- **`CurrentWParam` (new export).** Purely additive — a new symbol; no
  existing unit references it (verified: only `CurrentLParam` is grep-found,
  consumed by `StdDlgs.pas:82`). Cannot affect any other consumer. No
  `RepackWin32` change in the chosen design, so **no** other message and **no**
  other corpus app sees any behaviour change. (If the reviewer instead picks
  alternative (a), the new `RepackWin32` arm fires *only* for `WM_DROPFILES`
  $0233, which no other corpus handler reads from `Msg.LParam` — still
  contained.)
- **`DragQueryFile` signature widening.** Consumers of the symbol across the
  runtime + corpus:
  - `tests/corpus/auth/s3k_shellapi.pas` — only takes `@DragQueryFile`
    (address-of, link-presence check); never *calls* it. Unaffected by the
    parameter type.
  - `tests/corpus/ext/swan-bp7/.../WLIST/WLIST.PAS:478-492` — a real
    `WM_DROPFILES` handler that `uses ShellAPI` and passes a **literal
    `$FFFF`** count sentinel. Widening `iFile: Word → LongInt` is
    **source/compile backward-compatible**: a `$FFFF` literal fits a wider
    formal without error or new warning. WLIST keeps its own *latent*
    count-query + `Msg.WParam` truncation bug (identical to Nonogram's
    pre-fix state) — i.e. the widening does **not regress** WLIST; it merely
    does not fix it (out of scope). Note WLIST.PAS is **not** in the live bp7
    build manifest (`tests/common/bp7_manifest.rs` lists only `WLREG.PAS`),
    so it is not even a built/verified target — risk is minimal.
  - No other `runtime_pas` unit declares or calls `DragQueryFile`
    (`Win31.pas` does **not** — verified; the ShellAPI header's "duplicated
    with Win31" note concerns `ShellExecute` only).
- **Embedded-snapshot hash change.** Because both `.pas` files are
  `include_bytes!`-embedded, the FNV asset hash changes, so the rebuilt
  `mdtpw.exe` extracts to a *fresh* `%LOCALAPPDATA%\mdtpw\<hash>\` dir; prior
  caches (and thus any concurrently-running old builds) are left intact
  (`dist\README.md`). No cross-app contamination.

**Net blast radius:** additive runtime export + a strictly-more-correct,
source-compatible parameter widening. No behaviour change to any message or
corpus app other than Nonogram's drag-drop. The only theoretical risk is a
corpus program that *relied on* `DragQueryFile`'s `Word` truncation — none
exists.

## 7. Open design questions (for the reviewer)

1. **HDROP delivery mechanism.** Proposed: new `CurrentWParam` accessor
   reading the existing `DefWParam` latch (mirrors `CurrentLParam`; matches
   the STOPBUS `WMEraseBkgnd` precedent). Alternatives considered:
   `Msg.LParam` via a new `RepackWin32 WM_DROPFILES` arm (alt-a), or reading
   `DefWParam` directly (impossible — not exported). Confirm `CurrentWParam`
   is the preferred contract.
2. **`NumFiles` widening vs. explicit guard.** The sketch does *both* (widen
   to `LongInt` *and* `IF (NumFiles > 0)`). Widening alone already prevents
   the `FOR 0 TO -1` underflow. Keep both (defensive) or just the widening?
3. **Compiler rebuild sequencing.** Confirm route (A) — rebuild
   `mdtpw.exe` and reinstall to `C:\apps` — is acceptable (vs. dropping an
   in-tree `runtime_pas\` override beside the binary for a faster dev loop).
   `cargo build` of mdtpw **is** required either way for the embedded
   snapshot; `mdtpw_rt` floor needs no separate build.

## 8. Risk + rollback + commit messages

**Risk:** Low. The runtime change is additive (`CurrentWParam`) plus a
strictly-more-correct, source-compatible parameter widening; no `TMessage`
geometry, `RepackWin32` table, or external binding name changes. The largest
operational risk is the §0 sequencing trap — forgetting to rebuild/reinstall
`mdtpw.exe` so the app silently keeps the old embedded runtime. Mitigated by
making the rebuild+reinstall step 2/4 of the verify plan, and by the FNV hash
landing the new snapshot in a fresh cache dir (observable).

**Rollback:** `git revert` each commit independently. The two commits are
decoupled: the runtime commit is backward-compatible on its own (additive
export + wider-but-compatible param), and the app commit is inert until the
matching runtime is installed (it references `CurrentWParam`, which simply
won't link against an un-rebuilt runtime — a loud failure, not silent
corruption). Reverting the app commit alone restores the prior (broken but
building) `WMDropFiles`. Reverting the runtime commit alone requires also
reverting/installing the matching `mdtpw.exe`.

**Version bump:** the `runtime_pas\*.pas` files carry no version field; the
embedded runtime is versioned via the mdtpw crate. Bump
`C:\language\mdtpw\Cargo.toml` `version` `0.14.1 → 0.15.0` (minor — new public
`CurrentWParam` export + widened `DragQueryFile` signature are
backward-compatible additions). No app-side version field to bump
(`mdtpw.toml` has none).
