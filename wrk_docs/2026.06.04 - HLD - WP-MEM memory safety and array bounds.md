# HLD — WP-MEM: memory safety & array-bounds hardening (NONOGRAM.PAS)

**Document:** `2026.06.04 - HLD - WP-MEM memory safety and array bounds.md`
**Date:** 2026-06-04
**Branch:** `main`
**Work item:** WP-MEM — eliminate out-of-bounds array reads/writes (one heap-overflow
BLOCKER + several contained OOB defects) in the solver/editor, preserving 1994
behaviour except the defects.
**Status:** DESIGN ONLY — no code changes pending orchestrator sign-off.

---

## 1. Problem statement

`NONOGRAM.PAS` (6,255 lines, Borland TPW 1.5 / OWL, rebuilt 32-bit by `mdtpw
v0.14.1`) contains a family of unchecked array indexings that were latent at the
original 1994 limits (`MaxXAcross/MaxYAcross = 64`, `MaxNoElements = 15`) but are
now reachable, or already corrupt memory, after the port bumped the grid limits to
`100 / 100 / 40` (see the 2026-05-29 port HLD §3.1). One defect (A) writes past the
end of a `New`-allocated heap block and is a **BLOCKER**; the rest are contained
intra-object OOB reads/writes (B–F) plus a stale-documentation decision (G).

`INTEGER` is 16-bit. Arrays are 1-based as declared. The unit never sets `{$B}`, so
the Borland default `{$B-}` (short-circuit boolean evaluation) is in force across the
whole file — every fix below relies on putting the **bound test first** so the array
access is never evaluated when the index is out of range.

Baseline confirmed clean before any change:

```
mdtpw v0.14.1
  source  NONOGRAM.PAS  (6,255 lines)
  output  nonogram.exe  (250.5 KB)
  target  x32 windows PE32 O0
  time    0.19 s
```

## 2. Stakeholder & scope

Stakeholder: the person running `mdtpw` who wants a solver that does not corrupt the
heap or adjacent puzzle data on large boards or oversized `VARS` settings. Scope:
the seven WP-MEM findings below, all in `NONOGRAM.PAS`. **Out of scope:** any
gameplay/UX change, solver-algorithm rework, the resource/build pipeline, and
blanket `{$R+}` range-checking (see finding G).

## 3. Design

All line numbers verified against the working tree on 2026-06-04. Pascal sketches
match house style: UPPERCASE keywords, `(* … *)` comments, 2-space indent.

### 3.A BLOCKER — out-of-allocation heap overflow in `Combination`

**(1) Defect + location.** `NONOGRAM.PAS:3741` (X pass) and `NONOGRAM.PAS:3856`
(Y pass) write `PMainWindow(Parent)^.PotentData^.Data[A, PotenNum]` for
`A := 1..YAcross` (resp. `1..XAcross`), then `INC(PotenNum)` at `:3742` / `:3857`.
The only loop terminator is `IF (PotenNum > PMainWindow(Parent)^.VARSSize)` at
`:3745` / `:3860`. The 2nd dimension of `Data` is hard-coded:

```
NONOGRAM.PAS:175-177
  TPotVal = RECORD
    Data: ARRAY [1..MaxYAcross, 1..300] OF SHORTINT;
  END;
```

`PotentData` is one heap block, `New(PotentData)` at `NONOGRAM.PAS:4465`
(`100 × 300 × 1 byte = 30,000` bytes). `VARSSize` can exceed 300:

* the Solution-Options dialog `TConfigWin.OK` clamps `VARSSize <= 600`
  (`NONOGRAM.PAS:5631`);
* the INI loader applies **no** clamp:
  `VARSSize := GetPrivateProfileInt('Solution Options', 'VARS size', 300, INIFILENAME);`
  (`NONOGRAM.PAS:4528`).

**(2) Root cause.** `Data`'s 2nd dimension (300) and the `VARSSize` ceiling (600 in
the dialog, unbounded from INI) are two independent magic numbers that disagree.
With `VARSSize` in `301..600`, `PotenNum` reaches the dialog ceiling and the write
`Data[A, PotenNum]` (e.g. `Data[100, 600]`) lands ~`(600-300)*1 + row offset` ≈ up to
30,000 bytes past the 30,000-byte allocation → heap corruption. At the default
`VARSSize = 300` the array is exactly filled, so the bug is **latent** by default and
only triggers when a user raises VARS in the dialog or hand-edits `NONOGRAM.INI`.

**(3) Fix.** Introduce a single named constant and use it everywhere the `300`/`600`
magic numbers appear, **and** reorder so the bound is tested before the write.

*Insertion point — declarations, beside the existing limits at `NONOGRAM.PAS:167`:*

```
(* before *)
  MaxNoElements = 40;                        (* Maximum number of elements per row ... *)

(* after *)
  MaxNoElements = 40;                        (* Maximum number of elements per row ... *)
  MaxVARS = 600;                             (* Maximum VARS combinations stored per line in PotentData^.Data *)
```

*Record dimension — `NONOGRAM.PAS:176`:*

```
(* before *)
    Data: ARRAY [1..MaxYAcross, 1..300] OF SHORTINT;
(* after *)
    Data: ARRAY [1..MaxYAcross, 1..MaxVARS] OF SHORTINT;
```

*INI clamp — insert immediately after `NONOGRAM.PAS:4528`:*

```
(* before *)
  VARSSize := GetPrivateProfileInt('Solution Options', 'VARS size', 300, INIFILENAME);
(* after *)
  VARSSize := GetPrivateProfileInt('Solution Options', 'VARS size', 300, INIFILENAME);
  IF (VARSSize > MaxVARS) THEN VARSSize := MaxVARS;
  IF (VARSSize < 0) THEN VARSSize := 0;
```

*Dialog clamp — replace the magic `600` at `NONOGRAM.PAS:5631`:*

```
(* before *)
  IF (PMainWindow(Parent)^.VARSSize > 600) THEN PMainWindow(Parent)^.VARSSize := 600;
(* after *)
  IF (PMainWindow(Parent)^.VARSSize > MaxVARS) THEN PMainWindow(Parent)^.VARSSize := MaxVARS;
```

*Defence-in-depth (recommended) — reorder the store loop so the bound is checked
before the write, at both passes. X pass `NONOGRAM.PAS:3740-3749`:*

```
(* before *)
        IF (Flag = 0) THEN
        BEGIN
          FOR A := 1 TO YAcross DO
             PMainWindow(Parent)^.PotentData^.Data[A, PotenNum] := PossiblePos[A];
          INC(PotenNum);

          (* Check potential combinations number, PotenNum *)
          IF (PotenNum > PMainWindow(Parent)^.VARSSize) THEN
          BEGIN
            NotFinished := FALSE;
            More := TRUE;
          END;
        END;

(* after *)
        IF (Flag = 0) THEN
        BEGIN
          IF (PotenNum <= MaxVARS) THEN
            FOR A := 1 TO YAcross DO
               PMainWindow(Parent)^.PotentData^.Data[A, PotenNum] := PossiblePos[A];
          INC(PotenNum);

          (* Check potential combinations number, PotenNum *)
          IF (PotenNum > PMainWindow(Parent)^.VARSSize) THEN
          BEGIN
            NotFinished := FALSE;
            More := TRUE;
          END;
        END;
```

The identical guard is applied to the Y pass at `NONOGRAM.PAS:3853-3865` (loop is
`FOR A := 1 TO XAcross`). Because `VARSSize <= MaxVARS` after the clamps, this guard
never changes behaviour on a well-formed build; it is belt-and-braces so a future
ceiling change cannot re-introduce the overflow. (The downstream consensus read at
`:3795` iterates `B := 2 TO PotenNum-1` against `Data[A, B]`; with `VARSSize <=
MaxVARS` and `PotenNum` stopping at `VARSSize+1`, `B <= MaxVARS`, so it stays in
bounds. No change needed there.)

**(4) Alternatives + why rejected.**
* **`MaxVARS = 300`, clamp only, no array growth (RECOMMENDED ALTERNATIVE to
  consider).** Keeps `Data` at 30 KB; just clamp both sites to 300. Pro: zero heap
  growth, smallest diff. **Con: silently halves the user-facing VARS ceiling** the
  1994 dialog advertised (it accepts up to 600), changing behaviour for anyone who
  raised VARS to gain solving power on hard puzzles. That violates "preserve
  behaviour except the defect." **Rejected** in favour of `MaxVARS = 600`.
* **`MaxVARS = 600` (RECOMMENDED, chosen).** `Data` grows `30 KB → 60 KB`
  (`100 × 600 × 1`). Preserves the full advertised VARS range. The block is a single
  `New` on the heap (not the 64 KB-segment-bound stack/global of the 16-bit world),
  so 60 KB is trivial for the 32-bit PE. The `{$M 40000, 8192}` stack directive
  (`NONOGRAM.PAS:62`) is irrelevant — `Data` is heap, not stack. **Chosen.**
* **Dynamic sizing of `Data` to `VARSSize`.** Would need a pointer-to-open-array
  rewrite and reallocation on every options change — large blast radius, new failure
  modes, contrary to "smallest change." **Rejected.**
* **Clamp at the write site only (no shared constant).** Leaves the 300/600
  disagreement as live magic numbers; the next person who edits one forgets the
  other. **Rejected** — a named constant is the root-cause fix.

**(5) Blast radius / call sites.** `Data` is referenced only inside `Combination`
(`:3741, :3794, :3796, :3856`). `VARSSize` is read in `Combination` (`:3745, :3860`),
written in the INI loader (`:4528`) and `TConfigWin.OK` (`:5629-5632`), displayed in
`TConfigWin.SetupWindow` (`:5615`), and persisted on save. `MaxVARS` is new. No
public-API or file-format change; `NONOGRAM.INI` values >600 are now clamped down on
load (previously they corrupted memory). `New(PotentData)` size changes
transparently.

**(6) Test / verification.** Build: `mdtpw` in `C:\language\nonogram` (expect clean,
PE size grows ~30 KB). Manual proof of fix: set `[Solution Options] VARS size=600`
in `NONOGRAM.INI`, open a tall puzzle (e.g. `MILLENNI` 75×55) and run Solve — with
the old code this writes ~30 KB past the block (corruption / crash under a heap
checker); with the fix it completes. Confirm the dialog still accepts and round-trips
600. Regression: `solve-all.ps1` over all 51 `_puzzles\*.NGM` must remain
PASS/REJECTED-as-before with default `VARS size=300` (the default path is
byte-identical: `Data` indices 1..300 unchanged).

**(7) Rollback.** Revert the five hunks (constant, record dim, INI clamp, dialog
clamp, two store-loop guards). They are local and independent; reverting restores the
exact prior bytes.

### 3.B Issue — contained intra-object OOB write in `TEditer.OK`

**(1) Defect + location.** `NONOGRAM.PAS:5984` / `:5986` write
`PDisplay(Parent)^.XElements[NumberSelected, K]` / `YElements[...]` with `K := 1`
(`:5967`) and `INC(K)` (`:5989`) inside the token loop, with **no `K <=
MaxNoElements` guard**. `XElements`/`YElements` are `ARRAY[1..100, 1..40]`
(`NONOGRAM.PAS:290-291`).

**(2) Root cause.** Missing per-row element-count guard on user-typed clues. The edit
control is read by `Edit1^.GetLine(EditText, 100, 0)` (`:5946`) into
`EditText: ARRAY[0..100] OF CHAR` (`:556`, a field of the `TEditer` dialog), then a
`,` is appended (`:5947`). At most ~51 numeric tokens can fit in 100 chars, so K can
reach ~51 > 40, writing into `XElements[NumberSelected, 41..51]`, which (the array
being row-major `[1..100, 1..40]`) spills into the **next clue rows'** data — a
contained corruption inside the `TDisplay` object, not a heap-allocation overrun. The
analogous file-load path already guards this:

```
NONOGRAM.PAS:1937
  IF (LinePos+1 < MaxNoElements) THEN
    INC(LinePos)
  ELSE ... (* "Data file is corrupt" warning *)
```

**(3) Fix.** Mirror line 1937: stop converting once K reaches `MaxNoElements`.
Smallest form — guard the `INC` region so K never exceeds `MaxNoElements`.
*Insertion point — the token-commit block `NONOGRAM.PAS:5977-5991`:*

```
(* before *)
      IF (J > 0) THEN
      BEGIN
        (* Terminate string *)
        Spare[J] := #0;

        (* Convert string to a number and then copy it to parent field *)
        IF (UsingX) THEN
          Val(Spare, PDisplay(Parent)^.XElements[PDisplay(Parent)^.NumberSelected, K], Code)
        ELSE
          Val(Spare, PDisplay(Parent)^.YElements[PDisplay(Parent)^.NumberSelected, K], Code);

        (* Increment K and reset J *)
        INC(K);
        J := 0;
      END;

(* after *)
      IF ((J > 0) AND (K <= MaxNoElements)) THEN
      BEGIN
        (* Terminate string *)
        Spare[J] := #0;

        (* Convert string to a number and then copy it to parent field *)
        IF (UsingX) THEN
          Val(Spare, PDisplay(Parent)^.XElements[PDisplay(Parent)^.NumberSelected, K], Code)
        ELSE
          Val(Spare, PDisplay(Parent)^.YElements[PDisplay(Parent)^.NumberSelected, K], Code);

        (* Increment K and reset J *)
        INC(K);
        J := 0;
      END;
```

(Bound test first → safe under `{$B-}`. Leftover digits in `Spare` beyond the 40th
token are silently dropped, matching the "too many runs → ignore the excess"
behaviour the file loader already exhibits.)

**(4) Alternatives + why rejected.**
* **Surface the existing "Line entry truncated" / corrupt-data warning** like
  `:1941-1942` does on file load. Pro: user feedback. Con: more code, a status-bar
  dependency in a dialog OK handler, and a behaviour change (a message box / status
  text that did not exist before). **OPEN QUESTION for reviewer** — recommend silent
  truncation for the smallest change; add the warning only if desired.
* **Clamp `K` after `INC` instead of guarding the write.** Equivalent safety but
  reads less clearly and still evaluates the array index at K=41 before clamping
  unless ordered carefully. **Rejected** for clarity.

**(5) Blast radius / call sites.** `TEditer.OK` only. `XElements`/`YElements` for the
selected row are pre-cleared `1..MaxNoElements` just above (`:5957-5963`), so the
fix simply stops over-writing past 40. No format change.

**(6) Test / verification.** Build with `mdtpw`. Manual: open a clue editor field and
type >40 comma-separated numbers; with the fix only the first 40 are stored and the
following clue row is untouched (inspect via the editor on the next row). Regression:
`solve-all.ps1` unaffected (loads files, never exercises the typed editor).

**(7) Rollback.** Revert the single `IF` condition change.

### 3.C Issue — OOB read on all-zero rows in `FollowRow`

**(1) Defect + location.** `NONOGRAM.PAS:3371-3375` (X) and `:3500-3504` (Y):

```
NONOGRAM.PAS:3371-3375
  CurrentBlock := MaxNoElements;
  WHILE (XElements[I, CurrentBlock] = 0) DO
  BEGIN
    DEC(CurrentBlock);
  END;
```

No lower bound. A clue row that is entirely zero (an empty line — legitimate on a
sparse puzzle) decrements `CurrentBlock` below 1, reading `XElements[I, 0]`,
`[I, -1]`, … until it happens to hit a non-zero byte.

**(2) Root cause.** The loop assumes every row has at least one non-zero element;
empty rows violate that. Index test missing and placed after the access.

**(3) Fix.** Add the lower bound with the **index test first** (`{$B-}` safe), and
skip the row body when no block is found.
*X pass — `NONOGRAM.PAS:3371-3375`:*

```
(* before *)
      CurrentBlock := MaxNoElements;
      WHILE (XElements[I, CurrentBlock] = 0) DO
      BEGIN
        DEC(CurrentBlock);
      END;

(* after *)
      CurrentBlock := MaxNoElements;
      WHILE ((CurrentBlock >= 1) AND (XElements[I, CurrentBlock] = 0)) DO
      BEGIN
        DEC(CurrentBlock);
      END;
```

The Y pass at `NONOGRAM.PAS:3500-3504` takes the identical change against
`YElements`. The surrounding `WHILE (KeepGoing)` block (`:3377` / `:3506`) already
walks the grid; when `CurrentBlock = 0` the row has no blocks and the subsequent
`XElements[I, CurrentBlock]` accesses at `:3385`/`:3389` would read index 0 — so the
fix must also guard entry. Smallest robust form: wrap the existing per-row body in
`IF (CurrentBlock >= 1) THEN BEGIN … END;` so an empty row is skipped wholesale,
matching the original intent (an empty line has nothing to "follow"). Exact wrap
points: X body `:3377-…` (the `WHILE (KeepGoing)` block of this `FOR I` iteration),
Y body `:3506-…`. (Detailed brace placement to be finalised at code time; the
behaviour is "empty row ⇒ no-op", which is already what a correct follow yields.)

**(4) Alternatives + why rejected.**
* **`CurrentBlock := MaxNoElements - 1` start / sentinel scan upward.** Doesn't fix
  the all-zero case. **Rejected.**
* **Pre-count blocks with the `T`-counting idiom used in `Combination`
  (`:3709`).** Larger change, duplicates logic. **Rejected** — the two-line guard is
  sufficient.

**(5) Blast radius / call sites.** `FollowRow` only; gated by `RowFollowOn` (INI/
options). Empty-row puzzles previously read garbage and could mis-draw or crash; now
they are skipped. Non-empty rows: `CurrentBlock >= 1` always holds at the original
exit, so behaviour is byte-identical.

**(6) Test / verification.** Build with `mdtpw`. Manual: a puzzle with at least one
all-zero clue line, "Row following on", run Solve — no crash/garbage. Regression:
`solve-all.ps1`; library puzzles with blank lines exercise this path. Confirm no
PASS→FAIL deltas.

**(7) Rollback.** Revert the two `WHILE` conditions and the optional `IF` wrap.

### 3.D Issue — stale-size literals zero the wrong range in `ResetData`

**(1) Defect + location.** `NONOGRAM.PAS:4082-4085`:

```
  FOR I := 1 TO 25 DO
    YFilled[I] := 0;
  FOR I := 1 TO 20 DO
    XFilled[I] := 0;
```

`XFilled`/`YFilled` are `ARRAY[1..100] OF INTEGER` (`NONOGRAM.PAS:272-273`). The
literals `25`/`20` are pre-port grid sizes; rows `21..100` / `26..100` of the
filled-flag arrays are left **stale** by `ResetData`.

**(2) Root cause.** Hard-coded old dimensions never updated when the limits were
bumped to 100. (`ResetData` itself already uses `MaxYAcross`/`MaxNoElements`
correctly for the element/grid arrays at `:4071-4078` — only the filled arrays were
missed.)

**(3) Fix.** Use the named limits.
*`NONOGRAM.PAS:4082-4085`:*

```
(* before *)
  FOR I := 1 TO 25 DO
    YFilled[I] := 0;
  FOR I := 1 TO 20 DO
    XFilled[I] := 0;
(* after *)
  FOR I := 1 TO MaxYAcross DO
    YFilled[I] := 0;
  FOR I := 1 TO MaxXAcross DO
    XFilled[I] := 0;
```

**(4) Alternatives + why rejected.**
* **Do nothing — `CMSolve` re-clears per actual size at `:2427-2431`.** That masks
  the bug only on the *solve* path; `ResetData` is also called from `TDisplay.Done`
  (`:1306`), the file-load clear (`:1376`, `:1838`) and `CMNew` (`:6184`, `:6193`).
  After loading/creating a board taller than 25, stale `*Filled` flags from a prior
  puzzle can mark rows "already complete," suppressing solver work on them. **Not a
  no-op; rejected.**

**(5) Blast radius / call sites.** `ResetData` (`TDisplay.ResetData`, `:4061`), called
from `:1306, :1376, :1838, :6184, :6193`. The fix zeros 100 ints instead of 20/25 —
trivially cheap, strictly more-correct.

**(6) Test / verification.** Build with `mdtpw`. Manual: load a >25-row puzzle, then
File|New / load a different puzzle and Solve; rows beyond 25 must be re-solved (not
skipped as pre-filled). Regression: `solve-all.ps1` — tall puzzles (`MILLENNI`,
`PUMPKIN`) confirm rows >25 still solve identically.

**(7) Rollback.** Revert the two `FOR` bounds.

### 3.E Optional — OOB read when a clue row is full in `MakeText`

**(1) Defect + location.** `NONOGRAM.PAS:2968` (X) and `:2989` (Y):

```
  I := 1;
  WHILE (XElements[NumSel, I] > 0) DO
  BEGIN ... INC(I); END;
```

No `I <= MaxNoElements` bound. A row with all 40 elements non-zero reads
`XElements[NumSel, 41]` before the (non-existent) guard stops it.

**(2) Root cause.** Relies on a trailing zero terminator that a fully-populated row
(40 non-zero runs) lacks.

**(3) Fix.** Add the bound with **index test second is wrong**; put it first for
`{$B-}` safety:

```
(* before *)
    WHILE (XElements[NumSel, I] > 0) DO
(* after *)
    WHILE ((I <= MaxNoElements) AND (XElements[NumSel, I] > 0)) DO
```

Y pass `:2989` identical against `YElements`.

**(4) Alternatives + why rejected.** Reserve element 40 as a forced terminator —
changes data semantics and capacity. **Rejected.**

**(5) Blast radius / call sites.** `MakeText` (clue-string builder for the editor/
display). Only matters for rows with exactly `MaxNoElements` non-zero runs (rare but
reachable). Byte-identical for shorter rows.

**(6) Test / verification.** Build with `mdtpw`. Manual: a row with 40 runs, open its
editor — string builds without reading past the array. Regression: `solve-all.ps1`.

**(7) Rollback.** Revert the two `WHILE` conditions.

### 3.F Optional — 1-element OOB read from operand order under `{$B-}`

**(1) Defect + location.** Six sites use the idiom
`WHILE ((XElements[I, J] > 0) AND (J <= MaxNoElements))`:
`NONOGRAM.PAS:3075, :3092, :3132, :3161, :3936, :3953`. Because `{$B-}`
short-circuits left-to-right, the **array element is read before the bound test**, so
when `J = MaxNoElements + 1 = 41` the expression evaluates `XElements[I, 41]` (one
past the array) before the `J <= MaxNoElements` term can stop the loop.

**(2) Root cause.** Correct guard, wrong operand order for short-circuit evaluation.

**(3) Fix.** Swap the two operands at all six sites so the bound is tested first:

```
(* before *)
    WHILE ((XElements[I, J] > 0) AND (J <= MaxNoElements)) DO
(* after *)
    WHILE ((J <= MaxNoElements) AND (XElements[I, J] > 0)) DO
```

(`:3092, :3161, :3953` are the `YElements` twins — same swap.) Pure reordering of an
existing `AND`; no logic change for in-range J.

**(4) Alternatives + why rejected.** Enable `{$B+}` (full eval) locally — would
*hide* the read by evaluating both terms, i.e. it still reads `[I,41]`; the swap
actually prevents the read. **Rejected.**

**(5) Blast radius / call sites.** The six sites are in the parse/sum routines
(`ParseData`-family around `:3070-3165`) and a solver helper (`:3936/:3953`).
Identical results for valid data; the swap only changes the boundary case at J=41.

**(6) Test / verification.** Build with `mdtpw`. Regression: `solve-all.ps1` full
pass — these routines run on every load/solve, so a green batch proves no behaviour
change. (No dedicated puzzle needed; the change is provably equivalent for J ≤ 40.)

**(7) Rollback.** Revert the six conditions.

### 3.G Decision — the stale `{$R+}` claim in the header (NOT a code change)

**Finding.** The v2.3 history block claims the solver was made
`{$R+}`-range-check-clean (`NONOGRAM.PAS:41-42`: "…can be compiled in the `{$R+}`
state without getting run-time errors due to sloppy array handling"). Verified: **no
`{$R+}` directive exists anywhere in the file** — the only `{$R...}` token is the
disabled `{$R NONOGRAM.RES}` resource comment (`:58`); the only active range-relevant
directives are `{$I-}`/`{$I+}` around file IO (`:1841`/`:2022`). The claim is stale
(and the very defects in this WP show array handling is *not* `{$R+}`-clean).

**Recommendation.** Do **not** blanket-enable `{$R+}`. Rationale:
* `{$R+}` would add range checks across the *entire* unit, potentially tripping
  *other* latent index expressions we have not audited (e.g. the many `T+1`/sentinel
  walks), converting silent-but-currently-harmless reads into hard runtime traps —
  a broad, unpredictable behaviour change that violates "preserve behaviour except
  the defect" and risks surfacing issues outside WP-MEM's scope.
* The explicit per-site guards in A–F fix the actual defects precisely, with a
  reviewable blast radius, and cost nothing at runtime.

Instead, **correct the header** to reflect reality. Smallest honest edit: append a
note to the history block (this becomes the version-bump entry, see §6), e.g. a new
"Version 2.5" paragraph stating the array-bounds hardening was done with explicit
guards rather than `{$R+}`, and softening the 2.3 over-claim. No directive added.

**OPEN QUESTION for reviewer:** accept "explicit guards, no `{$R+}`" (recommended),
or require a `{$R+}` audit pass as separate future work?

## 4. Testability

* **Build oracle:** `mdtpw` must stay clean with no new warnings; PE grows ~30 KB
  (finding A's `Data` 30 KB→60 KB) and is otherwise unchanged.
* **Differential/regression oracle:** `solve-all.ps1` across all 51 `_puzzles\*.NGM`
  must report the *same* PASS/FAIL/REJECTED verdicts as the pre-change baseline. This
  is the primary safety net for B–F (and A at default VARS), since those routines run
  on every load/solve.
* **Targeted manual oracles** (one per finding, listed in each §3.x(6)): VARS=600 on
  `MILLENNI` (A); >40 typed runs (B); an all-zero clue line (C); load-tall-then-new
  then solve (D); a 40-run row in the editor (E); covered by regression (F).

## 5. Risk

* **Low–moderate.** The diff is small, local, and mechanical; every change either
  adds a short-circuit-safe bound test or substitutes a named constant for a magic
  number. The one structural item is the optional `IF` wrap in finding C (empty-row
  skip) — flagged as needing care at code time but behaviourally a no-op for
  non-empty rows.
* **Behaviour preservation:** default `VARS size=300` path is byte-identical; all
  bound tests only bite on out-of-range indices that previously corrupted memory or
  read garbage. The single deliberate behaviour change is INI `VARS size > 600` now
  clamping to 600 instead of corrupting the heap.
* **Residual:** findings E/F are marked optional; if descoped, the corresponding
  boundary reads (J=41 / I=41) remain latent but contained (single-element OOB read
  within the same global array, no allocation overrun). Recommend including them —
  they are one-line each.

## 6. Open design questions (for the reviewer)

1. **`MaxVARS = 600` vs `300`?** Recommend **600** to preserve the dialog's advertised
   range (Data 30 KB→60 KB heap, trivial on PE32). 300 is smaller-diff but silently
   halves a user capability.
