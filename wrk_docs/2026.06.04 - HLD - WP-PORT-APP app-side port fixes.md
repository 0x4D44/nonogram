## 0. Background — the runtime repack contract (verified facts)

Every finding below turns on how `RepackWin32` (runtime
`WObjects.pas:1795`) translates the raw Win32 `(AMsg, AWParam, ALParam)`
triple into the Win16-shaped `TMessage` the 1994 OWL handlers read. Verified
against `WObjects.pas`:

- **`TMessage` layout** (`WObjects.pas:302-309`): `WParam: Word` (16-bit),
  `LParam: LongInt` (32-bit, **not** truncated), with the Win16 half-word
  overlay `LParamLo, LParamHi: Word` aliasing `LParam`.
- **Default repack arm** (`WObjects.pas:1939-1941`):
  `Msg.WParam := LoWord(AWParam)` (truncates to 16 bits);
  `Msg.LParam := ALParam` (full 32-bit pass-through).
- **`WM_MDIACTIVATE` is NOT a case arm** in `RepackWin32` — it falls to the
  default arm above. So in `WMMDIActivate`, `Msg.WParam` is the truncated low
  word of the Win32 wParam and **`Msg.LParam` is the untouched 32-bit Win32
  lParam**. This is the linchpin of finding P+Q.
- **`wm_MenuSelect` arm** (`WObjects.pas:1890-1894`):
  `Msg.WParam := LoWord(AWParam)` (the 16-bit menu **item id**);
  `Msg.LParamLo := LoWord(ALParam)` (low word of the HMENU);
  `Msg.LParamHi := HiWord(AWParam)` (the Win32 **menu flags**, e.g.
  `MF_POPUP`). `Msg.WParam` is therefore **never** a menu handle; only the
  *low 16 bits* of the HMENU survive (in `LParamLo`), and the high word of the
  HMENU is **lost** (overwritten by the flags in `LParamHi`). Linchpin of
  finding T.
- **Callback trampoline width** (`WObjects.pas:2061`): `StdWndProc` /
  `DlgProc` / `DlgWindowProc` are all declared
  `(…; Msg: Longint; WParam, LParam: TWinParam): TWinResult`, where
  `TWinParam = TWinResult = LongInt` on Win32 / `Pointer` on Win64
  (`WinTypes.pas:800-801, 820-821`). A Pascal callback whose params are
  declared 16-bit `Word` makes the export trampoline narrow the native
  high word before the body sees it. Linchpin of finding S.
- **GetOpenFileName hook pass-through** (`CommDlg.pas:349-360`): the runtime
  wrapper strips `OFN_ENABLEHOOK` **only when `lpfnHook = nil`**. Nonogram
  sets `Ofn.lpfnHook := HookProc` (non-nil, `NONOGRAM.PAS:1810`), so the flag
  is **retained** and the real `GetOpenFileNameA` *will* call `HookProc` with
  native-width arguments. Confirms finding S is a live (not dormant) path.

No runtime change is proposed or required; all five fixes live in the app.

## 1. Finding P+Q — MDI menu state broken (PRIMARY)

### 1.1 Defect
- `NONOGRAM.PAS:1464` — `TDisplay.WMMDIActivate` decides which menu to show
  with `IF (BOOLEAN(Msg.WParam) = TRUE) THEN …`.
- `NONOGRAM.PAS:1465-1469` — both branches `PostMessage(… WM_MENUCHANGE, 0,
  MAKELONG(menuHandle, windowMenuHandle))`, packing two 16-bit handle halves
  into the lParam.
- `NONOGRAM.PAS:4771-4776` — `TMainWindow.WMMenuChange` now calls
  `TpwSetMenu(HWindow, NorMenu)` **unconditionally**, with a comment admitting
  the Win16 `MAKELONG` path is unusable.

Net symptom: when the **last** MDI child closes the frame never reverts to the
two-item "no document" `DEFMENU`; it is stuck on the full `NORMENU`. And the
`BOOLEAN(Msg.WParam)` test that *should* distinguish activate-vs-deactivate is
meaningless.

### 1.2 Root cause
On Win32 a child's `WM_MDIACTIVATE` carries `wParam = HWND being
DEACTIVATED`, `lParam = HWND being ACTIVATED` — **not** the Win16
`TRUE/FALSE`-in-low-word flag the 1994 code assumed. Because `WM_MDIACTIVATE`
hits the default repack arm (§0), `Msg.WParam` is `LoWord(deactivated HWND)`
— a meaningless 16-bit fragment, so `BOOLEAN(Msg.WParam)` is essentially
random. The `MAKELONG(...)` packed lParam loses the high 16 bits of each
32-bit Win32 menu handle, so even if `WMMenuChange` tried to unpack it, the
handles would be garbage — which is exactly why the ported `WMMenuChange` gave
up and hard-coded `NorMenu`. With `NorMenu` hard-coded, `DefMenu` is never
restored.

`WM_MENUCHANGE` here is the app's **private** message
(`WM_MENUCHANGE = WM_USER + 1`, `NONOGRAM.PAS:161`), posted from the child to
the frame and dispatched to `TMainWindow.WMMenuChange`
(`virtual WM_FIRST + WM_MENUCHANGE`, `:418-419`). Being a private message,
its wParam/lParam are entirely under the app's control — there is no Windows
contract to honour. That makes it the right channel to carry a 32-bit-safe
selector.

Menu storage (verified `NONOGRAM.PAS:4388-4391`):
```
NorMenu       := LoadMenu(HInstance, 'NORMENU');   { full menu }
DefMenu       := LoadMenu(HInstance, 'DEFMENU');   { no-document menu }
NorMenuWindow := GetSubMenu(NorMenu, 2);           { NORMENU's Window popup }
DefMenuWindow := GetSubMenu(DefMenu, 0);           { DEFMENU's File popup }
```
Both menus are loaded once at frame `Init` and live for the frame's lifetime
(destroyed at `:4481-4482`). `NorMenuWindow`/`DefMenuWindow` were the MDI
"Window" sub-menu handles the OWL `TMDIWindow` frame would hand to the
MDICLIENT so it could append the child-window list. (Note the `DefMenuWindow`
target is actually `DEFMENU`'s File popup — DEFMENU has no Window menu, so the
original treated its File popup as the nominal window-menu slot; DEFMENU has
no MDI child list, so this is inert.)

### 1.3 Fix

**Selector contract (the design's core decision).** Repurpose the private
`WM_MENUCHANGE` wParam as a 1-bit menu selector that survives 32-bit
truncation trivially (it is a small integer):

| `Msg.WParam` | meaning            | menu to install |
|--------------|--------------------|-----------------|
| `0`          | no open document   | `DefMenu`       |
| `1`          | document active    | `NorMenu`       |

lParam is unused (pass `0`), eliminating the dead `MAKELONG` packing.

**(a) `TDisplay.WMMDIActivate` — `NONOGRAM.PAS:1463-1469`.**

*Before:*
```pascal
  (* Update the menus *)
  IF (BOOLEAN(Msg.WParam) = TRUE) THEN
    PostMessage(PMainWindow(Parent)^.HWindow, WM_MENUCHANGE, 0,
                MAKELONG(PMainWindow(Parent)^.NorMenu, PMainWindow(Parent)^.NorMenuWindow))
  ELSE
    PostMessage(PMainWindow(Parent)^.HWindow, WM_MENUCHANGE, 0,
                MAKELONG(PMainWindow(Parent)^.DefMenu, PMainWindow(Parent)^.DefMenuWindow));
```

*After:*
```pascal
  (* Update the menus. On Win32 WM_MDIACTIVATE delivers the activated HWND in
     lParam (untruncated); we are activated when it equals our own HWindow.
     The desired menu is carried to the frame as a 32-bit-safe selector in
     WParam: 1 = document menu (NorMenu), 0 = no-document menu (DefMenu). *)
  IF (Msg.LParam = LongInt(HWindow)) THEN
    PostMessage(PMainWindow(Parent)^.HWindow, WM_MENUCHANGE, 1, 0)
  ELSE
    PostMessage(PMainWindow(Parent)^.HWindow, WM_MENUCHANGE, 0, 0);
```

**(b) `TMainWindow.WMMenuChange` — `NONOGRAM.PAS:4771-4776`.**

*Before:*
```pascal
PROCEDURE TMainWindow.WMMenuChange(VAR Msg: TMessage);
BEGIN
  (* Win32 menu handles cannot be packed through the Win16 MAKELONG path. *)
  TpwSetMenu(HWindow, NorMenu);
  DrawMenuBar(HWindow);
END;
```

*After:*
```pascal
PROCEDURE TMainWindow.WMMenuChange(VAR Msg: TMessage);
VAR
  NewMenu, NewWindowMenu: HMENU;
BEGIN
  (* Select the menu from the 32-bit-safe selector posted in WParam:
     1 = document active (NorMenu), 0 = no open document (DefMenu). *)
  IF (Msg.WParam = 1) THEN
  BEGIN
    NewMenu       := NorMenu;
    NewWindowMenu := NorMenuWindow;
  END
  ELSE
  BEGIN
    NewMenu       := DefMenu;
    NewWindowMenu := DefMenuWindow;
  END;

  (* Install the frame menu and tell the MDI client which popup is the
     Window menu so the child-window list is appended to the right place. *)
  TpwSetMenu(HWindow, NewMenu);
  SendMessage(ClientWnd^.HWindow, WM_MDISETMENU, 0,
              MAKELONG(NewWindowMenu, 0));
  DrawMenuBar(HWindow);
END;
```

**Insertion points:** in-place replacement of the two procedure bodies named
above. No declarations move; `NorMenu`/`DefMenu`/`NorMenuWindow`/
`DefMenuWindow` already exist as `TMainWindow` fields
(`NONOGRAM.PAS:398-399`), and `ClientWnd` is the live `PMDIClient`
(`:4404, 4873`).

**On `WM_MDISETMENU` (the one nuance).** The original Win16 code passed both
the frame menu *and* the window-menu handle through the posted message,
implying it wanted the MDICLIENT's window-menu kept in sync. The current
ported code dropped this entirely. Re-adding the `WM_MDISETMENU` send (which
takes the new *frame* menu in wParam and the new *window* menu in lParam under
Win16; under Win32 the frame menu is wParam and the window submenu is lParam)
keeps the MDI child-window list attached to the correct popup after a menu
swap. **Open design question Q1:** whether to include the `WM_MDISETMENU`
send at all. *Recommendation:* include it — it is the faithful behaviour, it
is what `NorMenuWindow`/`DefMenuWindow` exist for, and without it the Window
popup's child list can desync after a Nor→Def→Nor cycle. If the reviewer
prefers absolute-minimum change, the `SendMessage(... WM_MDISETMENU ...)` line
can be dropped and the two `*Window` fields left unused (matching today's
behaviour for the menu bar, just with Def/Nor now correctly selected). Both
sub-options are designed above; the menu-selection fix (the actual defect)
does not depend on the WM_MDISETMENU line.

### 1.4 Alternatives rejected
- **Unpack handles from the posted lParam (keep MAKELONG).** Rejected: 32-bit
  HMENUs do not fit two 16-bit `MAKELONG` halves — the very corruption that
  broke the port. A `MAKELPARAM`-of-pointers scheme would need pointer-width
  packing the private message can't express without storing handles in
  fields. The selector is strictly simpler and lossless.
- **Read the active HWND inside `WMMenuChange` via `MDIGETACTIVE` instead of
  passing a selector.** Rejected: races against the very activation in
  flight; the child already knows whether it is being activated, so the
  child is the correct authority. Smaller and race-free to pass a flag.
- **Drive the menu from `NoDisplayWin`/child count instead of
  WM_MDIACTIVATE.** Rejected: larger blast radius (touches the open/close
  bookkeeping) and changes more than the defect; `NoDisplayWin`
  (`:4890-4910`) already gates *toolbar* state, not the menu, and conflating
  them is scope creep.
- **`BOOLEAN(Msg.LParam <> 0)` instead of comparing to `HWindow`.** Rejected:
  `lParam<>0` is true whenever *any* child is activated, including the case
  where *another* child steals activation from us — we would post "activate
  my menu" on our own deactivation. Comparing to `HWindow` is the precise
  test.

### 1.5 Blast radius
- `TDisplay.WMMDIActivate` (child) and `TMainWindow.WMMenuChange` (frame)
  only. The private `WM_MENUCHANGE` message is posted **only** from
  `WMMDIActivate` (verified: grep shows the two posts at `:1465/:1468` are the
  sole senders), so changing its wParam/lParam contract is self-contained.
- `MAKELONG` now appears nowhere in the file after the edit (was only at
  `:1466/:1469`) — no shared helper depends on it.
- The `WM_MDISETMENU` send (if included) newly touches the MDICLIENT, but only
  with handles it already owns; on Win32 `DefFrameProc`/MDICLIENT honour
  `WM_MDISETMENU` natively.
- `NorMenuWindow`/`DefMenuWindow` transition from dead-since-port back to
  used; no new globals.

### 1.6 Test / verification
1. **Build:** run `mdtpw` in `C:\language\nonogram`; expect clean compile, no
   new warnings, `nonogram_mdtpw.exe` produced.
2. **Manual GUI — menu reverts (the defect):** launch; with no document the
   frame shows the 2-popup **File/Help** menu (DEFMENU). Open a `.NGM`
   (`File|Open`) → menu becomes the full **File/Options/Window/Help** menu
   (NORMENU). Close the last child → menu must revert to **File/Help**
   (DEFMENU). *Pre-fix this stays stuck on NORMENU; post-fix it reverts.*
3. **Manual GUI — multi-child no regression:** open two `.NGM`s; click between
   them; menu stays NORMENU throughout; the Window popup lists both children
   and reflects the active one (verifies the `WM_MDISETMENU` arm if included).
4. **Headless option:** the existing `*.ps1` File|Open driver can confirm the
   menu-bar item count via `GetMenuItemCount(GetMenu(hwnd))` before/after an
   open and after closing the last child (2 → N → 2).

### 1.7 Rollback
Revert the two procedure bodies to their pre-change text (single-file,
two-hunk revert). No resource, build-config, or runtime change to undo.

## 2. Finding T — popup status-bar hints dead (OPTIONAL, cosmetic)

### 2.1 Defect
`NONOGRAM.PAS:4741-4744` — `TMainWindow.WMMenuSelect` tests
`IF ((Msg.WParam = hMenu1) AND (hMenu1 <> 0)) THEN … 'File menu'` … through
`hMenu4`, where `hMenu1..4` are 32-bit `HMENU` from `GetSubMenu`
(`:4673-4692`). The intent is "when the user highlights a top-level popup
title, show e.g. *File menu* in the status bar." These comparisons never
fire, so the popup-title hints never appear.

### 2.2 Root cause
Per §0, the `wm_MenuSelect` repack arm sets `Msg.WParam := LoWord(item id)`
— a 16-bit menu *item id*, never a handle. So `Msg.WParam = hMenu1` (a 32-bit
handle) is never true. Worse, the full HMENU is **not** recoverable from
`Msg`: only `LoWord(HMENU)` survives in `Msg.LParamLo`, and `Msg.LParamHi`
holds the Win32 menu **flags** (not the HMENU high word). The Win16 idiom
"wParam is the popup's HMENU when MF_POPUP is set" simply does not hold under
the repack.

### 2.3 Fix — recommendation: **remove the dead code (Option B)**

The task offers two directions:

- **Option A (detect popup via flags + resolve submenu):** test
  `(Msg.LParamHi AND MF_POPUP) <> 0` to recognise a popup-title highlight,
  then map it to a name. *Problem:* having recognised a popup, we still need
  to know *which* popup. `Msg.WParam` for a popup highlight is the popup's
  **position index** (0,1,2,…) within its parent menu — not its id — but the
  position is ambiguous because the bar layout shifts (4 vs 5 popups depending
  on whether a child is maximised, exactly the case-analysis at
  `:4670-4693`). Reconstructing "is this File/Options/Window/Help" from a
  position index under that shifting layout reintroduces the same fragility,
  for status-bar text only.
- **Option B (remove):** delete the four `Msg.WParam = hMenuN` comparisons
  (`:4741-4744`) **and** the now-unused `hMenu1..4` machinery: the local
  `VAR hMenu1, hMenu2, hMenu3, hMenu4: HMENU` (`:4665`) and the three
  assignment blocks (`:4671-4693`) that populate them. The `CASE (Msg.WParam)`
  item-id hints (`:4697-4738`) are unaffected — those *do* work, because
  `Msg.WParam` is the item id and the cases compare against `CM_*`/`SC_*` item
  ids.

**Recommendation: Option B.** Justification: the popup-title hints are
cosmetic status-bar text; Option A cannot recover the information cleanly
under the verified repack and would add fragile position-index logic; Option B
*removes* code and dead state, aligning with "smallest change / least code"
and leaving the working item-id hints intact. **Open design question T1:**
confirm the reviewer accepts losing the four popup-title hints ("File menu" /
"Options menu" / "Window menu" / "Help menu") — this is the only behavioural
delta, and it is a sub-second transient string.

*Before (excerpt):*
```pascal
PROCEDURE TMainWindow.WMMenuSelect(VAR Msg: TMessage);
VAR
  hMenu1, hMenu2, hMenu3, hMenu4: HMENU;
BEGIN
  ...
  IF (GetSubMenu(GetMenu(HWindow), 4) = 0) THEN
  BEGIN
    hMenu1 := GetSubMenu(GetMenu(HWindow), 0); ...
  END;
  { + two more layout-dependent assignment blocks }
  ...
    IF ((Msg.WParam = hMenu1) AND (hMenu1 <> 0)) THEN StatbarHan^.SetText(0, 'File menu');
    IF ((Msg.WParam = hMenu2) AND (hMenu2 <> 0)) THEN StatbarHan^.SetText(0, 'Options menu');
    IF ((Msg.WParam = hMenu3) AND (hMenu3 <> 0)) THEN StatbarHan^.SetText(0, 'Window menu');
    IF ((Msg.WParam = hMenu4) AND (hMenu4 <> 0)) THEN StatbarHan^.SetText(0, 'Help menu');
  END;
END;
```

*After (excerpt):*
```pascal
PROCEDURE TMainWindow.WMMenuSelect(VAR Msg: TMessage);
BEGIN
  ...
  { item-id CASE block retained verbatim }
  END;
END;
```
i.e. drop the `VAR hMenu1..4` line, the three `hMenuN := GetSubMenu(...)`
blocks (`:4670-4693`), and the four popup comparisons (`:4741-4744`).

### 2.4 Alternatives rejected
- Option A — rejected above (fragile, cosmetic, *adds* code).
- Leave as-is — rejected: dead comparisons + dead local state are exactly the
  kind of confusing residue this campaign is clearing.

### 2.5 Blast radius
`TMainWindow.WMMenuSelect` only. The working `CM_*`/`SC_*`/window-list item-id
hints are untouched. `hMenu1..4` are method-local (not the field-level
`hMenu1` reused in `HandleOldFiles`/`HandleStatbar`, which are separate
locals) — removing them affects nothing else.

### 2.6 Test / verification
Build clean. Manual: hover menu **items** (File|Open, Options|Solve, etc.) →
correct item hints still show in the status bar (no regression). Hover the
popup **titles** → no hint (accepted cosmetic loss). No crash, no stuck text.

### 2.7 Rollback
Restore the deleted `VAR`, assignment blocks, and four comparisons
(single-file revert).

## 3. Finding S — latent callback width on HookProc (OPTIONAL)

### 3.1 Defect
`NONOGRAM.PAS:1774` —
`FUNCTION HookProc(HWindow: HWnd; Msg, wParam: WORD; lParam: LONGINT): WORD; FAR;`
— installed as the `OFN_ENABLEHOOK` callback at
`Ofn.lpfnHook := HookProc` (`:1810`) for `GetOpenFileName` (`:1813`).

### 3.2 Root cause
The mdtpw runtime's own window/dialog callbacks (`StdWndProc`,
`WObjects.pas:2061`) declare native pointer-width params
`(Msg: Longint; WParam, LParam: TWinParam): TWinResult` precisely so the
export trampoline does **not** truncate the Win32 high word. A callback
declared with 16-bit `Word` params (as `HookProc` is) makes the trampoline
narrow `Msg`/`wParam` to 16 bits before the body runs. Confirmed live (not
merely theoretical): the runtime `GetOpenFileName` wrapper only strips the
hook when `lpfnHook = nil` (`CommDlg.pas:351-352`); Nonogram supplies a
non-nil hook, so `GetOpenFileNameA` *does* invoke it with native-width args.
It is harmless **today** only because the body tests just
`IF (Msg = wm_InitDialog)` and returns `0` — `wm_InitDialog` ($0110) fits in
16 bits, and no high-word data is read. It is nonetheless the same defect
class as P+Q/T and should be brought to the runtime's callback-width pattern.

### 3.3 Fix
Match the `StdWndProc` signature shape: widen `Msg`/`wParam` to `LongInt`
(pointer-width-compatible on Win32; the runtime uses `TWinParam` but `LongInt`
is the Win32 realisation, and the app already imports plain Win16 types so
`LongInt` is the faithful minimal widening) and return a pointer-width result.

*Before:*
```pascal
FUNCTION HookProc(HWindow: HWnd; Msg, wParam: WORD; lParam: LONGINT): WORD; FAR;
```

*After:*
```pascal
FUNCTION HookProc(HWindow: HWnd; Msg, wParam: LONGINT; lParam: LONGINT): LONGINT; FAR;
```
Body unchanged (`IF (Msg = wm_InitDialog) THEN Ctl3dSubClassDlg(...);
HookProc := 0;`). The unused local `p: TFarProc` (`:1776`) can optionally be
dropped (it is dead) but that is outside this finding's defect; leave it to
avoid widening scope unless the reviewer wants the tidy.

**Insertion point:** the single declaration line `:1774`; the assignment at
`:1810` (`Ofn.lpfnHook := HookProc`) is unchanged. Verified `lpfnHook` is
typed `TCDHookProc = Pointer` (`CommDlg.pas:35, 62`), so a wider function
address still assigns cleanly (it was always a bare-address assignment).

### 3.4 Alternatives rejected
- **Set `lpfnHook := nil` and drop the hook entirely.** Rejected: would lose
  the `Ctl3dSubClassDlg` 3-D theming on the Open dialog — a behavioural change
  beyond the defect. The brief says preserve behaviour except the defect.
- **Leave as 16-bit (it works today).** Rejected: it is a latent
  high-word-truncation bug of the same class we are eliminating; widening is a
  one-line, zero-behaviour-change hardening.

### 3.5 Blast radius
One declaration line. No caller passes the result anywhere (the OS consumes
it); the body reads only `Msg` against a sub-$FFFF constant, so widening
cannot change current behaviour. `Ctl3d` units already in `USES`.

### 3.6 Test / verification
Build clean. Manual: `File|Open` → the common dialog appears with Ctl3D
shading (proves the hook still fires `wm_InitDialog`); pick a `.NGM` → it
loads (proves the dialog round-trips normally). No regression.

### 3.7 Rollback
Restore the one declaration line.

## 4. Finding V — version drift (OPTIONAL, project version touch)

### 4.1 Defect — three disagreeing version strings
- Header comment `NONOGRAM.PAS:54` → `Version  : 2.3`.
- `generated/nonogram.rc:82` ABOUTBOX static 554 → `"Version 2.3"`.
- `generated/nonogram.rc:233-234, 247, 252` VERSIONINFO →
  `FILEVERSION 2, 30, 0, 0` / `PRODUCTVERSION 2, 30, 0, 0` / `"2.30"`.
- `NONOGRAM.PAS:5439-5440` `TAboutBox.SetupWindow` →
  `Text5^.SetText('Version 2.4')`, `Text6^.SetText('Released: 28/12/1997')`.

So the running About dialog shows **2.4 / 28-12-1997**, file-properties shows
**2.30**, the static fallback text shows **2.3**, and the source header says
**2.3**.

### 4.2 Root cause
Authoring drift across 1994→1997: the runtime `SetText` at `:5439` is the
latest-touched surface (it overrides the static 554 text at dialog-setup, and
carries the 1997 release date), while the `.rc` and header were never
re-stamped. The static-554 `"Version 2.3"` text is *only ever a fallback* —
`SetupWindow` immediately overwrites it with `'Version 2.4'`, so the user
never sees 2.3 in the dialog (but does see 2.30 in Explorer's file
properties, which reads VERSIONINFO).

### 4.3 Fix — recommendation: canonicalise on **2.4** (released 28/12/1997)
The runtime-displayed `2.4 / 28/12/1997` is the latest authored intent (newest
content, has the real release date), so adopt it everywhere:

| Surface | File:line | Before | After |
|---------|-----------|--------|-------|
| Header comment | `NONOGRAM.PAS:53-54` | `Version : 2.3` (and `Date : 15/11/1994`) | `Version : 2.4` (keep authorship date or add a 28/12/1997 line — see Q) |
| ABOUTBOX static 554 | `nonogram.rc:82` | `"Version 2.3"` | `"Version 2.4"` |
| ABOUTBOX static 555 | `nonogram.rc:83` | `"Released: "` | leave (runtime sets the date); optional `"Released: 28/12/1997"` |
| VERSIONINFO FILEVERSION | `nonogram.rc:233` | `2, 30, 0, 0` | `2, 40, 0, 0` |
| VERSIONINFO PRODUCTVERSION | `nonogram.rc:234` | `2, 30, 0, 0` | `2, 40, 0, 0` |
| VERSIONINFO FileVersion str | `nonogram.rc:247` | `"2.30"` | `"2.40"` |
| VERSIONINFO ProductVersion str | `nonogram.rc:252` | `"2.30"` | `"2.40"` |
| `TAboutBox.SetupWindow` | `NONOGRAM.PAS:5439-5440` | `'Version 2.4'` / `'Released: 28/12/1997'` | **unchanged** (already canonical) |

(Numeric VERSIONINFO uses minor `40` to encode "2.4" consistently with the old
`30`="2.3" convention.) This is the project's single version touch for this
campaign (per the global version-management rule).

**Open design question V1:** confirm **2.4** (vs reverting the runtime to 2.3).
Recommendation 2.4 as above. Secondary Q: whether to update the header `Date`
field too, or only the `Version` field — recommend updating only `Version` to
2.4 and leaving the 1994 authorship date as historical (the release date lives
in the About dialog), to keep the header's meaning ("originally authored")
intact.

### 4.4 Alternatives rejected
- **Standardise on 2.3** (change the runtime `SetText` down to 2.3): rejected

- **Leave drift**: rejected — the campaign explicitly owns the version touch

## 6. Cross-cutting risk

- **Highest risk: P+Q** — it changes live message routing and the private
  `WM_MENUCHANGE` contract. Mitigated by: the message has a single sender
  (self-contained), the selector is a trivially-truncation-safe small int, and
  the `WM_MDISETMENU` line is optional/severable so the core menu-selection
  fix can ship without it. Verification is a concrete, observable GUI cycle
  (Def → Nor → Def on last-child-close).
- **Lowest risk: V, S, licence-cleanup** — metadata / one-line widening / dead
  code removal, all behaviour-preserving (S is provably no-op today; V doesn't
  branch any code; licence-cleanup removes only blank/dead surface).
- **T** carries the only *intended* behaviour delta (four cosmetic popup-title
  hints lost) — gated behind reviewer sign-off (Q T1).
- **No runtime change** in any finding — the repack table and trampolines are
  treated as fixed contracts; if a future M/N runtime package adds a
  `WM_MDIACTIVATE` repack arm, the P+Q fix's reliance on the *default* arm
  (LParam pass-through) remains valid as long as `WM_MDIACTIVATE` lParam stays
  untruncated (it is `LongInt`, so it always will).
- **Build:** all `.rc` edits flow through the existing pinned `rc.exe` step
  in `mdtpw.toml`; no new build steps. Re-run `mdtpw`; require zero new
  warnings (house rule).
- **Version touch:** finding V is the campaign's single semantic-version bump
  (2.3/2.30 → 2.4/2.40), applied consistently across header, `.rc` static, and
  `.rc` VERSIONINFO.
