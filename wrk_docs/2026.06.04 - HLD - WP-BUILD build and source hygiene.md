# HLD — WP-BUILD: build output naming & source hygiene

**Document:** `2026.06.04 - HLD - WP-BUILD build and source hygiene.md`
**Date:** 2026-06-04
**Branch:** `main`
**Work item:** Stop the 32-bit `mdtpw` build from clobbering the tracked
1990s 16-bit `NONOGRAM.EXE`, restore the destroyed historical binary, and
remove the wrong-file hazard from three stale pre-port source variants.
**Touches no `NONOGRAM.PAS` code** — build config, a tracked binary blob, and
repo-hygiene files only.

---

## 1. Problem statement

The 32-bit rebuild is wired so that every `mdtpw` invocation silently
overwrites the tracked historical 16-bit `NONOGRAM.EXE`, and that overwrite
has **already been committed** — the genuine 16-bit New Executable (NE) is
gone from the `NONOGRAM.EXE` slot in `HEAD`. Separately, three stale
pre-port `.pas` copies in the repo root invite an edit (or an `mdtpw src=`
re-point / restore) into the wrong file, which would silently reintroduce the
old 64×64 / ≤15-run limits and the licence gate.

Two findings:

- **Finding U (BLOCKER, data loss, already committed):** `mdtpw.toml` emits
  `nonogram.exe`, which on case-insensitive Windows *is* the tracked
  `NONOGRAM.EXE`. The build clobbers it and the clobber is in `HEAD`.
- **Finding W (optional, wrong-file hazard):** `nonogram_work.pas`,
  `nonogram_survey.pas`, `nonogram.pas.orig` sit in the root in the OLD
  pre-port state.

## 2. Stakeholder & scope

Stakeholder: the person in the nonogram repo who expects `mdtpw` to build the
32-bit GUI **without** destroying the preserved 1990s artifacts, and who
expects edits to land in the one canonical source. Scope: `mdtpw.toml`,
`.gitignore`, the tracked `NONOGRAM.EXE` blob, and the three stale `.pas`
variants. **Out of scope:** any `NONOGRAM.PAS` source change, the solver, the
resource pipeline, and all generated artifacts (`*.ll`, `*.obj`,
`nonogram.pas.orig` qua generated junk is handled only as a stale-source
variant, not as a build output).

## 3. Ground truth (verified)

All facts below were confirmed by inspecting git and decoding the PE/NE
headers (read `e_lfanew` at offset `0x3C`, then the signature there):

| Object | Size | `e_lfanew` | Sig | Blob hash |
|---|---|---|---|---|
| `HEAD:NONOGRAM.EXE` | 248 832 | `0x78` | **PE** | `8d6566e…` |
| `HEAD:WIN16/NONOGRAM.EXE` | 117 248 | `0x100` | **NE** | `584ab38…` |
| `5f2dd47~1:NONOGRAM.EXE` (parent) | 117 248 | `0x100` | **NE** | `584ab38…` |
| working-tree `nonogram.exe` (on disk, lowercase) | 256 512 | — | **PE** | (newer build, untracked-as-such) |

Key derived facts:

- `core.ignorecase=true` (verified). The on-disk file is literally named
  `nonogram.exe` (lowercase) but resolves to the same inode git tracks as
  `NONOGRAM.EXE`. `git status` reports `M NONOGRAM.EXE`.
- `git check-ignore nonogram.exe` → **exit 1 (NOT ignored)**;
  `git check-ignore nonogram_mdtpw.exe` → **exit 0 (ignored)**. So the build
  output name `nonogram.exe` is tracked and re-clobbers every build, while the
  *intended* name `nonogram_mdtpw.exe` is already correctly ignored.
- The genuine NE survives, byte-identical, at **two** reachable points:
  `WIN16/NONOGRAM.EXE` (blob `584ab38…`) and `5f2dd47~1:NONOGRAM.EXE` (same
  blob `584ab38…`). The clobber happened at commit `5f2dd47`.
- The index already holds the PE blob (`git ls-files -s NONOGRAM.EXE` →
  `8d6566e…`), confirming the loss is committed, not merely working-tree.
- The three stale variants are all in the OLD state and all git-ignored:

| File | Lines | `MaxXAcross` | `MaxNoElements` | `CheckLicence` | `{$R NONOGRAM.RES}` | ignored |
|---|---|---|---|---|---|---|
| `NONOGRAM.PAS` (canonical) | 6255 | **100** | **40** | absent | disabled (comment L58) | tracked |
| `nonogram_work.pas` | 6369 | 64 | 15 | present | active | yes |
| `nonogram.pas.orig` | 6369 | 64 | 15 | present | active | yes |
| `nonogram_survey.pas` | 6169 | 64 | 15 | present | active | yes |

- No tracked file references the stale variants except `.gitignore` and the
  prior HLD (both descriptive, not load-bearing). `_survey/` exists and is
  git-ignored.

## 4. Finding U — build clobbers the tracked 16-bit `NONOGRAM.EXE` (BLOCKER)

### (1) Confirmed defect + exact location
`C:\language\nonogram\mdtpw.toml:19`
```toml
output    = "nonogram.exe"
```
This contradicts the file's own comment (`mdtpw.toml:13-14`: "Output:
`nonogram_mdtpw.exe`, deliberately suffixed so the build never clobbers the
tracked 1990s 16-bit NONOGRAM.EXE") and the `.gitignore:21` rule
(`nonogram_mdtpw.exe`). Because `core.ignorecase=true`, `nonogram.exe`
collides with the tracked `NONOGRAM.EXE`; the build overwrites it and the
overwrite is committed in `HEAD` (`HEAD:NONOGRAM.EXE` is now PE/248 832B,
not NE/117 248B).

### (2) Root cause
A single wrong string at `mdtpw.toml:19`. The whole hygiene design (comment +
`.gitignore`) assumes the output is `nonogram_mdtpw.exe`; only the active
`output` key was left at the colliding name, defeating the entire scheme. The
case-insensitive filesystem turns "different name" into "same file".

### (3) Proposed fix (concrete before/after + exact location)

**3a. Rename the build output — `mdtpw.toml:19`**
```toml
# before
output    = "nonogram.exe"
# after
output    = "nonogram_mdtpw.exe"
```
This is the one functional change that stops the clobber: `nonogram_mdtpw.exe`
is already in `.gitignore:21`, so future builds write an ignored file and the
tracked slot is never touched again.

**3b. Restore the historical NE into the tracked `NONOGRAM.EXE` slot.**
The build currently leaves a PE in `NONOGRAM.EXE` (working tree + index +
HEAD). Restore the NE blob `584ab38…` from the surviving copy. Exact commands
(run in `C:\language\nonogram`):
```powershell
# Recover the original NE from the byte-identical surviving blob.
git checkout 5f2dd47~1 -- NONOGRAM.EXE
# (equivalent source, same blob 584ab38…:  git checkout HEAD -- WIN16/NONOGRAM.EXE  then copy)
```
Then **verify the restored blob is the NE** before staging/committing:
```powershell
# size must be 117248; first PE/NE sig must be "NE"
(Get-Item .\NONOGRAM.EXE).Length                       # -> 117248
git hash-object .\NONOGRAM.EXE                          # -> 584ab38140a8438162dec6d6eb338f9008e225a2
# signature check (read e_lfanew @0x3C, then 2 bytes there):
$b=[IO.File]::ReadAllBytes("$pwd\NONOGRAM.EXE"); $o=[BitConverter]::ToInt32($b,0x3C)
[Text.Encoding]::ASCII.GetString($b,$o,2)              # -> NE
```
The restore is exact because `git hash-object` of the restored file must equal
`584ab38140a8438162dec6d6eb338f9008e225a2`; any other hash means the wrong
blob and the commit must not proceed.

**3c. Make name usage consistent and add a defensive ignore — `.gitignore`.**
The `.gitignore` already ignores `nonogram_mdtpw.exe` (line 21) — keep it.
Add a defensive guard so a future mis-set `output` can never *re-track* the
collision name, while still keeping the historical `NONOGRAM.EXE` tracked
(git keeps tracking files already in the index regardless of ignore rules):
```gitignore
# before  (.gitignore:19-21)
# The build EXE is deliberately suffixed so it never clobbers the tracked 1990s
# 16-bit NONOGRAM.EXE (see mdtpw.toml `output`).
nonogram_mdtpw.exe
# after
# The build EXE is deliberately suffixed so it never clobbers the tracked 1990s
# 16-bit NONOGRAM.EXE (see mdtpw.toml `output`). The lowercase `nonogram.exe`
# is also ignored defensively: on case-insensitive Windows it aliases the
# tracked NONOGRAM.EXE, so a stray build with that name must never re-track it.
nonogram_mdtpw.exe
nonogram.exe
```
`mdtpw.toml:13-14`'s comment is already correct and needs no edit.

This satisfies the open question "whether to also add `nonogram.exe` to
`.gitignore` defensively" — **recommend yes**: it is belt-and-braces against
the exact failure that already occurred, costs nothing, and does not untrack
the historical `NONOGRAM.EXE` (already in the index). Note: it *will* keep the
current stray working-tree `nonogram.exe` ignored, which is the desired
outcome once the slot holds the restored NE.

### (4) Alternatives + why rejected
- **Add `nonogram.exe` to `.gitignore` but leave `output` as-is.** Rejected:
  ignore rules do not apply to already-tracked files, so the build would still
  overwrite the tracked `NONOGRAM.EXE` working-tree file every time and it
  would still show as modified / get committed. Fixing the name at the source
  (`output`) is the actual fix.
- **Untrack `NONOGRAM.EXE` entirely (`git rm --cached`) and rely on `WIN16/`.**
  Rejected: changes the repo's preservation contract (the root `NONOGRAM.EXE`
  is the documented historical artifact per `.gitignore:5-7`); out of scope
  and a larger blast radius than the request.
- **Restore from `WIN16\NONOGRAM.EXE` by file copy instead of
  `git checkout 5f2dd47~1`.** Equivalent (identical blob `584ab38…`), but the
  `git checkout <rev> -- path` form is self-documenting about provenance and
  avoids a manual copy step. Either is acceptable; commands for both are
  listed. Not rejected — offered as the documented equivalent.
- **Rename to some other suffix (e.g. `nonogram32.exe`).** Rejected: the repo
  already standardised on `nonogram_mdtpw.exe` (`.gitignore`, the comment, the
  prior HLD §3.3); introducing a third name re-opens the inconsistency.

### (5) Blast radius
- `mdtpw.toml:19` — changes only the emitted filename. Any script or doc that
  launches the build output by hard-coded name must use `nonogram_mdtpw.exe`.
  The headless verification scripts (`*.ps1`, `solve-all.ps1`) already target
  `nonogram_mdtpw.exe` per the prior HLD §4 — **must be re-confirmed** during
  test (see §6).
- `NONOGRAM.EXE` blob restore — touches one tracked binary; no source/build
  dependency reads it (the build doesn't consume it). Pure preservation fix.
- `.gitignore` — adds one ignore line; only affects an untracked
  `nonogram.exe`, never the tracked `NONOGRAM.EXE` in the index.

### (6) Test & verification plan
1. **Pre-restore proof of defect:** `git ls-files -s NONOGRAM.EXE` →
   `8d6566e…` (PE), and the signature check yields `PE`/248 832B. (Already
   confirmed.)
2. **Apply 3a/3b/3c.**
3. **Restore proof:** after 3b, `git hash-object NONOGRAM.EXE` →
   `584ab38140a8438162dec6d6eb338f9008e225a2`; size `117248`; sig `NE`. This is
   the oracle — the SHA-1 equality is a complete proof the bytes are the
   original NE.
4. **Build proof:** run `mdtpw` in `C:\language\nonogram` (~1s). Expect it to
   emit `nonogram_mdtpw.exe` and **not** modify `NONOGRAM.EXE`. Verify with
   `git status --porcelain NONOGRAM.EXE` → **empty** (clean) after a build,
   and `git status --porcelain nonogram_mdtpw.exe` → empty too (ignored).
5. **No-regression:** run `solve-all.ps1` against the 51-puzzle corpus; expect
   the same PASS/FAIL/REJECTED profile as before (the binary is functionally
   identical — only its filename changed). Confirm the scripts launch
   `nonogram_mdtpw.exe`; if any still reference `nonogram.exe`, that is a
   follow-up to repoint (flagged as an open question).
6. **Hygiene proof:** `git check-ignore nonogram.exe` → exit 0 after 3c;
   `git check-ignore NONOGRAM.EXE` → exit 1 *but* `git ls-files NONOGRAM.EXE`
   still lists it (tracked despite ignore — desired).

### (7) Rollback
All three sub-changes are independently revertible:
- `mdtpw.toml:19` and `.gitignore` — `git checkout -- mdtpw.toml .gitignore`.
- `NONOGRAM.EXE` blob — `git checkout HEAD -- NONOGRAM.EXE` restores the
  pre-fix (PE) state; no data is lost because the NE is permanently reachable
  at `WIN16/NONOGRAM.EXE` and `5f2dd47~1`. The fix only ever *adds* safety.

## 5. Finding W — three stale pre-port `.pas` variants in the root (optional)

### (1) Confirmed defect + exact location
`C:\language\nonogram\nonogram_work.pas` (6369 L),
`C:\language\nonogram\nonogram.pas.orig` (6369 L),
`C:\language\nonogram\nonogram_survey.pas` (6169 L). All three are in the OLD
pre-port state: `MaxXAcross = 64`, `MaxNoElements = 15`, `CheckLicence`
present, `{$R NONOGRAM.RES}` active (line numbers in §3 table). All three are
git-ignored (`git check-ignore` exit 0 for each).

### (2) Root cause
Leftover survey/working copies from the port effort, never cleared from the
working tree. They are not a build hazard *today* (`mdtpw.toml:18` reads
`NONOGRAM.PAS`), but they are a latent wrong-file hazard: an edit, an
`mdtpw src=` re-point, or a "restore from the .orig" would silently
reintroduce the 64×64/≤15 limits and the licence gate.

### (3) Proposed fix — relocate into the already-ignored `_survey\`
Move the three files out of the root into the existing git-ignored scratch dir
`_survey\` (which already holds survey copies). Exact commands (in
`C:\language\nonogram`):
```powershell
Move-Item .\nonogram_work.pas    .\_survey\nonogram_work.pas
Move-Item .\nonogram.pas.orig    .\_survey\nonogram.pas.orig
# _survey\nonogram_survey.pas already exists and is identical-in-kind; either
# overwrite it or drop the root copy:
Move-Item -Force .\nonogram_survey.pas .\_survey\nonogram_survey.pas
```
Because both source and destination are git-ignored, **no `git mv` and no
commit is required** — this is a pure working-tree tidy with zero index/history
effect. The `.gitignore:23-26` rules naming these three files in the root can
optionally be left as-is (harmless) or trimmed; **recommend leaving them** so a
future stray root copy stays ignored. (Documenting this as the chosen path
under "Repo hygiene" in the next commit message.)

### (4) Alternatives + why rejected
- **Delete from the working tree.** Safe (git-ignored ⇒ no history loss;
  `nonogram.pas.orig`'s content is also recoverable as the pre-port source via
  history, and `nonogram_survey.pas` already exists in `_survey\`). Rejected as
  the *default* only because relocate is strictly more conservative: it keeps
  the bytes one move away if anyone still wants to diff against the OLD state,
  at no extra cost. **Recommendation: relocate.** If Arthur prefers a clean
  root with nothing extra, delete is equally acceptable and I'll take that
  steer (open question).
- **Add a `{$IFDEF}` / header banner to each stale file warning "NOT CANONICAL".**
  Rejected: edits ignored scratch files for no durable benefit; relocation
  removes the hazard structurally instead of relying on a human reading a
  banner.
- **Leave them in place.** Rejected: that is the status quo hazard the finding
  exists to remove.

### (5) Blast radius
None on the tracked tree or build: the files are git-ignored and unreferenced
by any tracked, load-bearing file (verified: only `.gitignore` and the prior
HLD mention them, descriptively). `mdtpw` reads `NONOGRAM.PAS` only.

### (6) Test & verification plan
1. After the move: `git status --porcelain` shows **no change** (both ends
   ignored) — proves zero tracked-tree impact.
2. `Test-Path .\nonogram_work.pas, .\nonogram.pas.orig, .\nonogram_survey.pas`
   → all `False`; the three now exist under `_survey\`.
3. `mdtpw` still builds (reads `NONOGRAM.PAS`); `solve-all.ps1` unchanged.

### (7) Rollback
`Move-Item .\_survey\<file> .\<file>` restores the prior layout; or, since the
files are git-ignored, simply leave them in `_survey\`. No history to revert.

## 6. Overall risk

- **Finding U:** Low-risk fix, high-value. The functional change is one string
  (`output`); the binary restore is proven exact by SHA-1 equality
  (`584ab38…`). The only residual risk is a verification script still hard-coded
  to `nonogram.exe`; mitigated by §4(6) step 5. **This is the blocker and
  should land first.**
- **Finding W:** Negligible risk (git-ignored files, pure working-tree move).
  Optional; can land in the same commit or be deferred.

No new compiler warnings are possible (no `NONOGRAM.PAS` change). `mdtpw.toml`
has no version field to bump; if a versioned file is touched in
implementation, apply a **patch** bump per house rules.

## 7. Open design questions for the reviewer

1. **Restored-binary naming/versioning:** confirm the tracked root artifact
   stays `NONOGRAM.EXE` holding the original NE (blob `584ab38…`), and that
   `WIN16\NONOGRAM.EXE` remains the redundant safe copy. (No version field on a
   binary; nothing to bump.)
2. **Delete vs relocate (Finding W):** recommendation is *relocate* into
   `_survey\`; *delete* is equally safe. Which do you want?
3. **Defensive `.gitignore` for `nonogram.exe`:** recommendation is *add it*
   (belt-and-braces against the exact recurrence). Confirm.
4. **Verification-script audit:** do `solve-all.ps1` / the `*.ps1` drivers
   already launch `nonogram_mdtpw.exe`? If any reference `nonogram.exe`, a
   follow-up repoint is needed (out of WP-BUILD scope, but flagging).
