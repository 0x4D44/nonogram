# HLD — NONOGRAM 16-bit → 32-bit port (mdtpw toolchain)

**Document:** `2026.05.29 - HLD - NONOGRAM 16-bit to 32-bit mdtpw port.md`
**Date:** 2026-05-29
**Branch:** `main`
**Work item:** Rebuild the 1993-97 Nonogram app as a native 32-bit PE with mdtpw.

---

## 1. Problem statement

`NONOGRAM.PAS` is a ~6,300-line Borland Turbo Pascal for Windows (TPW 1.5)
OWL application — a nonogram (griddler) editor and solver, shipped 1993-1997
as a 16-bit Windows binary (`NONOGRAM.EXE`). The goal is to rebuild it
unmodified-in-spirit as a native 32-bit PE using the in-house `mdtpw`
toolchain (Pascal → LLVM IR → COFF → `lld-link` → PE32), the same path
already proven on the sibling `stopbus` project.

Three things stand between the 1990s source and a modern PE32:

1. **A commercial licence gate.** Startup was guarded by `CheckLicence`,
   which read `NONOGRAM.LIC` with Turbo Pascal typed-file I/O
   (`FILE OF CHAR`) and a byte-obfuscation scheme. It is dead weight for a
   personal rebuild and additionally trips the compiler (see §3.1).
2. **Grid size limits too small for the bundled puzzles.** Compile-time
   constants capped the board at 64×64 with ≤15 runs per line; the puzzle
   library includes boards up to 75×55 (MILLENNI) with 15-16 run lines.
3. **An incompatible resource format.** The 16-bit `NONOGRAM.RES` is the
   Win16 record format; `lld-link` accepts only Win32 `.RES`.

## 2. Stakeholder & scope

Stakeholder: the person sitting in the nonogram repo who wants to type
`mdtpw` and get a working 32-bit GUI build. Scope: source edits required to
compile + run correctly, the resource pipeline, the `mdtpw.toml` build
config, and repo hygiene. **Out of scope:** any gameplay/UX change, solver
algorithm rework, or touching the historical 16-bit artifacts beyond keeping
them tracked.

## 3. Design

### 3.1 Source changes (`NONOGRAM.PAS`, +8 / -233)

- **Strip the licence gate.** Remove `FindBytes` and `CheckLicence`
  (~233 lines) and start the app directly:
  `NonogramApp.Init('Nonogram'); NonogramApp.Run; NonogramApp.Done;`
  instead of `IF CheckLicence THEN …`. This is desired on its own merits
  *and* removes the `FILE OF CHAR` typed-file construct that the mdtpw
  parser rejected (`build_log.txt`: "expected `;` after var declaration" at
  the `FileVar: FILE OF CHAR;` declaration). The `UserName` / `CompanyName`
  vars and their About-box display are retained (harmless, no longer gated).
- **Raise grid limits.** `MaxXAcross` / `MaxYAcross` 64 → 100 (fits MILLENNI
  75×55 and matches the app's own ≤100 load guard); `MaxNoElements` 15 → 40
  (fits the 15-16 run lines in PUMPKIN/MILLENNI).
- **Copyright glyph byte** in the About box (encoding tidy-up).

No solver, UI, or file-format logic was changed.

### 3.2 Resource pipeline

The 16-bit `NONOGRAM.RES` stays tracked as the historical artifact. The
modern Win32 resource is `generated/nonogram.rc` plus its `.bmp`/`.ico`
assets, reconstructed from the 16-bit `.RES` by mdtpw's `nonores` tool
(`cargo run --bin nonores`). `mdtpw.toml` lists `generated/nonogram.rc` under
`resources`; the build pipeline auto-compiles it to a Win32 `.res` via the
pinned SDK `rc.exe` and links it. `generated/` is therefore a tracked build
*input*, not an output. This mirrors `stopbus` (`generated/stopbus.rc`).

### 3.3 Build config (`mdtpw.toml`)

```toml
[package]
name      = "nonogram"
src       = "NONOGRAM.PAS"
output    = "nonogram_mdtpw.exe"   # suffixed: never clobber the 16-bit NONOGRAM.EXE
subsystem = "windows"
resources = ["generated/nonogram.rc"]
```

`output` is deliberately `nonogram_mdtpw.exe` so the rebuild never overwrites
the tracked 1990s `NONOGRAM.EXE` (an improvement over stopbus, whose build
clobbers `STOPBUS.EXE`).

### 3.4 Repo hygiene (`.gitignore`)

Tracked: the originals (`NONOGRAM.PAS`, `NONOGRAM.RES`, `NONOGRAM.EXE`,
`RELEASE/`, `HELP/`, installer), the modern build inputs (`mdtpw.toml`,
`generated/`), the puzzle library (`_puzzles/`), the headless verification
scripts (`*.ps1`), and these working docs. Ignored: build output
(`*.ll`, `*.obj`, `nonogram_mdtpw.exe`), scratch source copies
(`nonogram.pas.orig`, `nonogram_work.pas`, `nonogram_survey.pas`), the 16-bit
resource backup (`NONOGRAM.RES.orig16`), runtime config (`NONOGRAM.INI`),
logs, and the scratch dirs (`_diag/`, `_survey/`, `shots/`).

## 4. Testability

Correctness is verified end-to-end through the GUI, headlessly: the `*.ps1`
scripts launch `nonogram_mdtpw.exe`, drive File|Open + Solve via
`PostMessage`, capture the rendered grid with `PrintWindow` (works in a
headless session where a desktop grab is blank), classify each cell, and run
an independent run-length clue-check oracle against the `.NGM`. `solve-all.ps1`
batches this across the whole puzzle library and reports PASS/FAIL/REJECTED.
See the verification journal for the PUMPKIN (solves) and MILLENNI (provably
unsatisfiable) results.
