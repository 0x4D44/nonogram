## 1. Finding H — NIL-deref crash in `CheckGroup` progress update

### 1.1 Defect + location
`NONOGRAM.PAS:2745`

```pascal
PMainWindow(Parent)^.StatbarHan^.SetText(0, @T1);
```

This is the **only** unguarded `StatbarHan^.SetText` caller in the file. Every
other caller (47 sites; e.g. 1433, 1849, 1978, 2406, 2450, 2482, 2577, 2799,
5483) is wrapped in `IF (PMainWindow(Parent)^.StatbarExist) THEN …`.

### 1.2 Root cause
`StatbarHan: PStatbar` is initialised `NIL` (TMainWindow.Init, line 4410) and
set back to `NIL` when the status bar is torn down (`HandleStatbar`,
lines 4846–4850: `DestroyWindow(StatbarHan^.HWindow); StatbarHan := NIL;
StatbarExist := FALSE;`). With the status bar turned off via Preferences,
`StatbarHan = NIL`. The line dereferences `NIL^.SetText` → access violation.

**Reachability (verified):** Preferences → turn status bar off (sets
`StatbarExist := FALSE`, `StatbarHan := NIL`). Then Options | Bad data recovery
→ `CM_RECOVER` → `TDisplay.CMRecover` → `CheckGroup`. The crash line sits in
the `LowestPos` progress branch (2739–2746), hit whenever recovery descends
into a new lowest group position — i.e. on essentially any non-trivial recover.

### 1.3 Fix — before / after
Insertion point: wrap line 2745. The surrounding block (2739–2746):

**Before**
```pascal
      IF (Group < LowestPos) THEN
      BEGIN
        LowestPos := Group;
        StrCopy(T1, 'Current lowest position:');
        Str(LowestPos, T2);
        StrCat(T1, T2);
        PMainWindow(Parent)^.StatbarHan^.SetText(0, @T1);
      END;
```

**After**
```pascal
      IF (Group < LowestPos) THEN
      BEGIN
        LowestPos := Group;
        StrCopy(T1, 'Current lowest position:');
        Str(LowestPos, T2);
        StrCat(T1, T2);
        IF (PMainWindow(Parent)^.StatbarExist) THEN
          PMainWindow(Parent)^.StatbarHan^.SetText(0, @T1);
      END;
```

Matches the established house pattern exactly (cf. 2577, 2799). 2-space indent,
UPPERCASE keywords.

### 1.4 Alternatives + why rejected
- **Guard on `StatbarHan <> NIL` instead of `StatbarExist`** — functionally
  equivalent (the two are always kept in lockstep) but inconsistent with the 47
  existing sites. Rejected for style/consistency.
- **Initialise a dummy/null-object status bar** — large surface change, premature
  abstraction, violates smallest-change. Rejected.

### 1.5 Blast radius
One statement, gated by a flag that is already maintained correctly. When the
status bar exists, behaviour is byte-identical (the `IF` is true). When it does
not, the progress text is silently skipped — exactly as all other progress
updates already behave. No effect on the recovery algorithm or solver.

### 1.6 Test / verification
- Build: `mdtpw` → exit 0, no new warnings.
- **Manual proof:** load a puzzle, Preferences → status bar OFF, Options | Bad
  data recovery on a puzzle that descends groups. Pre-fix: AV crash. Post-fix:
  recovery runs to completion with no status text. Repeat with status bar ON →
  the "Current lowest position:" text appears as before.
- **Regression:** `solve-all.ps1` — all 51 puzzles still PASS (this path is not
  exercised by a normal solve; the guard cannot change solve output).

### 1.7 Rollback
Revert the two added lines (the `IF … THEN` wrapper) at 2745. Self-contained.

## 2. Finding I — copy-paste wrong-axis guard in `FollowRow` 4th pass

### 2.1 Defect + location
`NONOGRAM.PAS:3493`

```pascal
  (* Work from XAcross downwards in the YACross direction *)
  FOR I := 1 TO YAcross DO
  BEGIN
    (* If the row is not already full then... *)
    IF (XFilled[I] <> 1) THEN          (* <-- wrong array *)
```

### 2.2 Root cause
`FollowRow` has four passes. Verified the 4th pass body (3490–3554) is entirely
**Y-direction**: the loop is `FOR I := 1 TO YAcross`, and the body reads
`YElements[I, CurrentBlock]` (3514) and `ScreenGrid[J, I]` (3508, 3533, 3544) —
identical orientation to the 3rd pass (3428–3487), whose guard correctly reads
`YFilled[I]` (3431). The 4th pass's guard was copy-pasted from an X-pass and the
`XFilled` → `YFilled` rename was missed.

`XFilled` is declared `ARRAY [1..MaxXAcross]`, `YFilled` `ARRAY [1..MaxYAcross]`
(lines 272–273; `MaxXAcross = MaxYAcross = 100` post-port). Indexing
`XFilled[I]` with `I` up to `YAcross` is in-bounds (both arrays are 100 wide),
so this is **not** a memory bug — it is a *logic* bug: the pass decides whether
to skip a Y-row by consulting the X-row "already full" flag of the same index.

**Effect:** when row `I` differs in fullness between axes, the backward Y-pass
either (a) re-processes an already-full Y-row (`YFilled[I]=1` but `XFilled[I]≠1`)
— wasteful but the inner `DrawPoint` re-asserts identical cells, or (b) **skips a
not-yet-full Y-row** (`YFilled[I]≠1` but `XFilled[I]=1`) — this is the harmful
case: the backward edge-deduction for that row is silently dropped, so the
solver may converge slower or, in principle, miss a deduction the forward pass
also missed. It does not produce *wrong* cells (every cell `FollowRow` draws is
forced), but it weakens the pass.

### 2.3 Fix — before / after
Single-token change at 3493:

**Before**
```pascal
    IF (XFilled[I] <> 1) THEN
```
**After**
```pascal
    IF (YFilled[I] <> 1) THEN
```

### 2.4 Alternatives + why rejected
None reasonable — this is an unambiguous one-character rename to match the pass
orientation. Any "alternative" would be leaving the bug in place.

### 2.5 Blast radius
One identifier. Only affects which Y-rows the backward FollowRow pass processes.
Because every cell FollowRow asserts is logically forced, output remains a subset
of the true solution; this fix can only *add* correct deductions or *remove*
redundant re-draws, never introduce a wrong cell. Solve results are unchanged or
strictly improved.

### 2.6 Test / verification
- Build: `mdtpw` → exit 0.
- **Regression is the proof:** `solve-all.ps1` must still report all 51 puzzles
  PASS (the oracle independently clue-checks every solved grid). Because the fix
  can only add forced cells, no puzzle that solved before can fail after.
- **Targeted manual check:** a puzzle that relies on FollowRow edge-completion
  on a tall/asymmetric board (e.g. MILLENNI 75×55) — confirm it still solves /
  rejects identically. Optionally toggle Options so only Row-following is enabled
  to isolate the pass.

### 2.7 Rollback
Revert `YFilled` → `XFilled` at 3493.

## 3. Finding J — cooperative-multitasking reentrancy during a solve

### 3.1 Defect + location
`Yielder` (`NONOGRAM.PAS:2566–2583`) pumps `PeekMessage`/`TranslateMessage`/
`DispatchMessage` mid-solve and special-cases **only** `CM_STOPSO`
(2571–2579: sets `SolveInProgress := FALSE`, `CanThink := FALSE`). **All other**
messages — including state-mutating `WM_COMMAND`s — are dispatched normally
while a `Combination` / `FollowRow` / `CheckEdge` frame is suspended on the stack.

### 3.2 Root cause — architecture (verified)
The solve runs in `TDisplay.CMSolve` (2399+) on the **active MDI child**. Its
crunch loop (2472–2493) calls the algorithm methods, which call `Yielder`. A
`Yielder` dispatch can re-enter a command handler that mutates the very
`TDisplay` instance whose solve frame is suspended.

Critical routing fact: **`CM_LOAD` is bound on BOTH classes** —
`TDisplay.CMLoad` (decl 319, body 1786, `virtual CM_FIRST + CM_LOAD`) and
`TMainWindow.CMLoad` (decl 430, body 5061). In OWL MDI command routing the
**active child handles the command first**; since `TDisplay` *has* a `CMLoad`,
`CM_LOAD` dispatched mid-solve runs **`TDisplay.CMLoad` on the solving window
itself** → opens a **modal** GetOpenFileName, then `FileLoadImplement` (1822) →
`ResetData` (1838 → 4061) zeroes `XElements`/`YElements`/`ScreenGrid` and
reassigns `XAcross`/`YAcross` (1885–1894) **on the same object** while the
suspended `Combination` frame still holds live indices into them
(`T`, `Start[1..T]`, the `WHILE NotFinished` enumeration over `XElements[I,…]`).
Result: corrupt enumeration, wrong/garbage cells, or out-of-range index.

The other state-mutating entries are **frame-only** (`TMainWindow`):
- `CMNew` (5023) → creates a *new* child + grays menus/toolbar.
- `CMLoad` (5061) → `FileOpener` → *new* child + `FileLoadImplement` on it.
- `CMFile1..4` (5113–5138) → `FileOpener` (MRU) → *new* child.
- `WMDropFiles` (4749, `virtual WM_FIRST + WM_DROPFILES`) → `FileOpener` per
  dropped file → *new* child(ren).
These do **not** call `ResetData` on the solving child, but they DO, mid-solve:
reassign the frame's `DisplayHan` to a brand-new child (5030/5227), gray the
solve/stop menu items (5041–5044/5241–5244) and toolbar buttons while the old
solve is still running, and (CMNew/FileOpener) call `Application^.MakeWindow`
re-entrantly. The post-solve epilogue (2503–2510) then restores menus/title for
the *old* child against a frame whose `DisplayHan` now points elsewhere — UI
desync and a second live "solvable" window with the menu in the wrong state.

**Already guarded:** window CLOSE — `TDisplay.CanClose` (1346–1359) and
`TMainWindow.CanClose` (5277–5290) defer via `CloseAfterSolve` /
`StopSolAndExit` / `ChildClosing`. State-mutating *commands* are not guarded.

### 3.3 Design decision — RECOMMENDED: per-handler early-EXIT gate

**Recommendation: gate each state-mutating command handler to early-EXIT (with
a `MessageBeep(0)`) when a solve is in progress** — rather than filtering
non-`CM_STOPSO` `WM_COMMAND`s inside `Yielder`.

Rationale (why per-handler over a Yielder filter):
1. **Yielder runs on the child (`TDisplay`); the dangerous handlers are split
   across both classes.** `TDisplay.Yielder` would have to swallow commands
   destined for the *frame* (`TMainWindow`) too. Identifying "is this a
   state-mutating command" by `wParam` inside Yielder duplicates the routing
   knowledge and is fragile against future menu additions — exactly the
   "premature abstraction / road to nowhere" the house style warns against.
2. **Swallowing in Yielder also eats the menu's normal feedback** (e.g.
   `WMMenuSelect` status-bar text) and any benign command, changing behaviour
   on paths we don't intend to touch. The per-handler gate touches *only* the
   handlers that are actually unsafe.
3. **The guard is local and self-documenting** at each unsafe entry, mirroring
   the existing `IF (SolveInProgress)` style already used in `CanClose`
   (1349, 5285).
4. **Smallest correct change** to the not-solving path: when
   `SolveInProgress = FALSE`, every guarded handler runs exactly as today.

Which `SolveInProgress` to test:
- For `TDisplay.CMLoad`: the child's own `SolveInProgress` field (1349 already
  uses the bare `SolveInProgress` in this class) **OR** the frame's
  `PMainWindow(Parent)^.SolveInProgress`. Both are set true together in
  `CMSolve` (2416–2417) and false together (2500–2501). Use the frame field for
  uniformity with the frame handlers — `PMainWindow(Parent)^.SolveInProgress`.
- For the `TMainWindow` handlers: the frame's own `SolveInProgress` (bare, as in
  `CanClose` 5285).

### 3.4 Exact handlers to gate + guard sketch

**Complete list of state-mutating entries reachable via a mid-solve `Yielder`
dispatch** (verified — these are every command/message that creates a child,
reassigns `XAcross/YAcross`, or calls `ResetData`/`FileLoadImplement`):

| # | Handler | File:line | Path | Frame field to test |
|---|---------|-----------|------|---------------------|
| 1 | `TDisplay.CMLoad`      | 1786 | self → `FileLoadImplement` → `ResetData` | `PMainWindow(Parent)^.SolveInProgress` |
| 2 | `TMainWindow.CMNew`    | 5023 | new child + menu/toolbar grays | `SolveInProgress` (own) |
| 3 | `TMainWindow.CMLoad`   | 5061 | `FileOpener` → new child | `SolveInProgress` (own) |

### 3.5 Alternatives + why rejected
- **Filter non-`CM_STOPSO` `WM_COMMAND`s inside `Yielder`** (the stated
  alternative): rejected per §3.3 — wrong class to own the policy, duplicates
  routing knowledge, eats benign commands and menu feedback, fragile to future
  menu items.
- **Disable (gray) all File menu items + `DragAcceptFiles(HWindow, FALSE)` for
  the solve duration:** more code, must be unwound on every exit path of
  `CMSolve` (including the interrupt and the error EXIT at 2412), and does not
  cover a command already queued before graying. The early-EXIT gate is
  immune to queued commands and needs no unwind. Rejected.
- **A global reentrancy counter / critical section:** over-engineered for a
  cooperative single-threaded message pump. Rejected.

### 3.6 Blast radius
Eight handlers gain a 4-line prologue that is a strict no-op when
`SolveInProgress = FALSE` (the normal case). The not-solving path is unchanged.
Mid-solve, these commands now beep-and-ignore instead of corrupting state — a
strict safety improvement with no effect on the solver's output for any puzzle
that is solved normally (no command interleaving). `solve-all.ps1` drives
File|Open then Solve sequentially (open completes before solve starts), so the
gate is never tripped by the regression harness.

### 3.7 Test / verification
- Build: `mdtpw` → exit 0.
- **Manual proof (the core bug):** load a large slow puzzle (e.g. MILLENNI),
  start Solve, and **while it is solving** invoke File | Open (and separately:
  File | New, an MRU entry, and drag-drop a `.NGM`). Pre-fix: state corruption /
  crash / UI desync. Post-fix: a beep, the command is ignored, the solve
  continues and completes normally. CM_STOPSO still aborts.
- **Regression:** `solve-all.ps1` — all 51 PASS (harness never interleaves, so
  identical results; confirms the not-solving path is untouched).

### 3.8 Rollback
Remove the prologue from each of the eight handlers. Each is independent;
partial rollback is safe.

## 4. Finding K — non-interruptible greedy enumeration in `Combination`

### 4.1 Defect + location
`NONOGRAM.PAS:3719` (X pass) and `3834` (Y twin):
```pascal
IF NOT(PMainWindow(Parent)^.GreedyModeOn) THEN Yielder;
```
inside the inner `WHILE (NotFinished)` arrangement-enumeration loop (3717–3787 /
3832–3902). With greedy mode ON, the only `Yielder` is **once per row**
(3697 / 3812). The inner enumeration of a single wide row can run a very large
number of iterations with **no** message pump, so `CM_STOPSO` cannot interrupt
it — worse since the grid cap rose 64 → 100.

### 4.2 Root cause
The author deliberately suppressed yielding in greedy mode to maximise
throughput, but tied the *interruptibility* check to the same yield. In greedy
mode the inner loop neither yields nor consults `CanThink`, so once it enters a
long row it is committed to enumerating that row to completion (or until
`PotenNum > VARSSize` trips `More`). `CanThink` is only re-read at the *outer*
`FOR I` row guard (3700 / 3815), never inside the inner `WHILE`.

### 4.3 Fix — before / after
Add a cheap iteration counter that pumps `Yielder` every `N` inner iterations
**even in greedy mode**, and break the inner loop when `CanThink` goes false.
This keeps greedy fast (one pump per N, not per iteration) while restoring
interruptibility and bounded latency.

Insertion: declare a counter in the existing `VAR` block (3681–3689). Reuse one
new local, e.g. `YieldCnt: INTEGER;` (note: INTEGER is 16-bit, so the threshold
must keep the counter well inside ±32767 — `N` small, counter reset each pump).

**Before (X pass, 3717–3720):**
```pascal
      WHILE (NotFinished) DO
      BEGIN
        IF NOT(PMainWindow(Parent)^.GreedyModeOn) THEN Yielder;
```
**After:**
```pascal
      WHILE (NotFinished AND PMainWindow(Parent)^.CanThink) DO
      BEGIN
        IF NOT(PMainWindow(Parent)^.GreedyModeOn) THEN
          Yielder
        ELSE
        BEGIN
          INC(YieldCnt);
          IF (YieldCnt >= GreedyYieldEvery) THEN
          BEGIN
            YieldCnt := 0;
            Yielder;
          END;
        END;
```
Initialise `YieldCnt := 0;` immediately before each inner `WHILE` (i.e. just
after `NotFinished := TRUE;` at 3705 / 3820). Apply the **identical** change to
the Y twin at 3834 (`WHILE (NotFinished)` at 3832 → add `AND
PMainWindow(Parent)^.CanThink`).

Adding `AND PMainWindow(Parent)^.CanThink` to the `WHILE` condition is what
actually *breaks out* once `Yielder` (via a dispatched `CM_STOPSO`) clears
`CanThink`. With `{$B-}` short-circuit eval in force, `CanThink` is only read
after `NotFinished` is true — cheap and safe.

### 4.4 Threshold design (`GreedyYieldEvery`)
Add a CONST near the other tuning constants (top of unit, ~line 82+):
```pascal
GreedyYieldEvery = 2048;   (* Pump messages every N greedy inner iterations *)
```
Rationale: 2048 inner iterations is negligible CPU on a modern machine
(sub-millisecond), giving worst-case interrupt latency far below human
perception, while the pump cost (one `PeekMessage` returning false) is amortised
to ~0.05% overhead — greedy stays effectively as fast as today. The value is a
power of two (cheap mental model) and comfortably inside 16-bit INTEGER range,
so `INC(YieldCnt)` cannot overflow before the reset at `>= 2048`.

### 4.5 Termination & correctness argument
- The inner `WHILE` still terminates by its original logic: `NotFinished` is
  cleared by the block-rearrangement exhaustion (3778–3782) or the `More` cap
  (3745–3749). Adding `AND CanThink` can only make it terminate **earlier**,
  never later — no new non-termination path.
- When interrupted mid-row, `More` is still `FALSE` (the cap wasn't hit), so the
  post-loop `IF NOT(More)` draw block (3790–3802) runs on the partial
  `PotentData` gathered so far. **This is identical to today's non-greedy
  interrupt behaviour** — Yielder can already clear `CanThink` between non-greedy
  inner iterations, leaving the same partial state. So interrupting greedy now
  behaves exactly as interrupting non-greedy already does: a partial,
  still-correct (forced-cells-only) grid, then the outer loop's `CanThink` guard
  stops further rows. No wrong cells: every drawn cell is the intersection of all
  enumerated valid arrangements (3792–3800), which remains sound on a prefix.
- **When NOT interrupted, output is byte-identical to today:** the only added
  behaviour is periodic `Yielder` calls that, absent a `CM_STOPSO`, do not touch
  any solver state (they pump benign messages; with finding J in place, the
  dangerous commands are gated). The enumeration proceeds to the same completion.

### 4.6 Alternatives + why rejected
- **Check `CanThink` only (no extra Yielder) in greedy:** insufficient — without
  a pump, `CM_STOPSO` never reaches the queue, so `CanThink` never flips. Must
  pump. Rejected.
- **Yield every iteration in greedy (delete the greedy special-case):** removes
  the entire performance point of greedy mode; defeats the user's setting.
  Rejected.
- **Time-based pump (GetTickCount every iteration):** `GetTickCount` per
  iteration is more costly than a counter compare and adds a syscall to the hot
  loop. The counter is simpler and cheaper. Rejected.

### 4.7 Blast radius
Two inner loops (X + Y), one new CONST, one new local per call. Greedy
throughput essentially unchanged; non-greedy path unchanged (still yields every
iteration). Solver output unchanged on every non-interrupted solve.

### 4.8 Test / verification
- Build: `mdtpw` → exit 0.
- **Regression (primary):** `solve-all.ps1` — all 51 PASS. Greedy is the default
  solve mode the harness uses; if the counted-yield perturbed enumeration, a
  puzzle would FAIL. All must still PASS → proves output unchanged.
- **Manual interrupt proof:** enable Greedy (Preferences), load the widest/slowest
  puzzle, Solve, then Stop solving mid-row. Pre-fix: Stop is ignored until the
  row's full enumeration completes (visible lag). Post-fix: Stop takes effect
  within ~one pump interval; status shows "Solution interrupted".
- **Timing sanity:** confirm greedy solve time on MILLENNI/PUMPKIN is within
  noise of the pre-fix time (no measurable regression).

### 4.9 Rollback
Restore the two `WHILE (NotFinished) DO` conditions and the two
`IF NOT(GreedyModeOn) THEN Yielder;` lines; remove the CONST and the `YieldCnt`
local + its init. Self-contained.

## 5. Finding L — 16-bit overflow in `CalComplexity` (display-only)

### 5.1 Defect + location
`CalComplexity` (`NONOGRAM.PAS:4180–4377`):
```pascal
VAR
  A: ARRAY [1..1000] OF INTEGER;                 (* 4182 *)
  ...
  BlockTot, Basenum, Numblk, Totpos, TotComb: INTEGER;   (* 4184 *)
```
The recurrence main loop `A[J] := A[J]+A[K]` (4293 / 4359) grows
super-exponentially; `TotComb` accumulates these (4280/4299/4346/4365). Both
`A[]` elements and `TotComb` are 16-bit `INTEGER` → wrap on wide rows. The
author's own clamp confirms the known overflow:
```pascal
  IF (CombinX < 0) THEN CombinX := 0;            (* 4375 *)
  IF (CombinY < 0) THEN CombinY := 0;            (* 4376 *)
```
(`CombinX/CombinY` are already `LONGINT`, fields 263–264, but they accumulate an
already-wrapped 16-bit `TotComb`, so widening them alone — as the existing code
shows — does not help.)

### 5.2 Root cause
Combinatorial counts exceed 32,767 on large boards; 16-bit `A[]`/`TotComb`
silently wrap, producing garbage (often negative) totals.

### 5.3 Impact (verified display-only)
`CombinX/CombinY` are consumed **only** by `TInfoBox.SetupWindow`
(6073–6099): the "average combinations" number (6074–6082) and the textual
complexity level (6085–6099). **The solver never reads them** — `CMSolve`
(2399+) operates solely on `XElements/YElements/ScreenGrid`. So this defect
corrupts only the Information dialog's displayed numbers, never a solve.

### 5.4 Fix — before / after
Widen `A[]` and `TotComb` (and the intermediates that feed them) to `LONGINT`,
and **saturate** at a sentinel when growth would overflow, so the displayed
total caps cleanly instead of wrapping.

**Before (decls 4182–4184):**
```pascal
  A: ARRAY [1..1000] OF INTEGER;
  I, II, J, K, M: INTEGER;
  BlockTot, Basenum, Numblk, Totpos, TotComb: INTEGER;
```
**After:**
```pascal
  A: ARRAY [1..1000] OF LONGINT;
  I, II, J, K, M: INTEGER;
  BlockTot, Numblk, Totpos: INTEGER;
  Basenum, TotComb: LONGINT;
```
(`Basenum` widened too — it seeds `A[I] := Basenum-I` and `TotComb := Basenum`;
`BlockTot/Numblk/Totpos` are bounded by grid size ≤100 and stay 16-bit; loop
indices `I/J/K/M` stay INTEGER, but note `FOR I := 1 TO (Basenum-1)` now iterates
a LONGINT bound — acceptable since the real iteration count never approaches
32767 before saturation caps it; see below.)

**Saturation** — define a sentinel CONST near the top:
```pascal
MaxComplexity = 1000000000;   (* Saturation cap for displayed combinations *)
```
and clamp at each accumulation. Replace the two summation spots
(4279–4280 / 4298–4299 and the Y twins 4345–4346 / 4364–4365):
```pascal
      FOR I := 1 TO (Basenum-1) DO
      BEGIN
        TotComb := TotComb + A[I];
        IF (TotComb > MaxComplexity) THEN TotComb := MaxComplexity;
      END;
```
and clamp `A[J]` in the recurrence (4293 / 4359):
```pascal
            IF (K = J) THEN
              A[J] := A[K]
            ELSE
            BEGIN
              A[J] := A[J] + A[K];
              IF (A[J] > MaxComplexity) THEN A[J] := MaxComplexity;
            END;
```
Capping `A[]` keeps the inner `FOR I := 1 TO (Basenum-1)` bound (`Basenum` is
derived from grid size, ≤ ~100, so the loop count is tiny) and prevents the
LONGINT itself from overflowing. With the cap, `CombinX/CombinY` accumulate
saturated LONGINTs; the existing `< 0` clamps (4375–4376) become dead but stay
(harmless, smallest change).

### 5.5 "No other code depends on 16-bit" — verified
- `A[]` and `TotComb` are **locals** of `CalComplexity`; no external dependency.
- `Basenum` likewise local.
- `CombinX/CombinY` are already `LONGINT`; consumers (6074–6079) do `DIV K`,
  `+`, `Str(...)` — all `LONGINT`-clean. `Str(LONGINT)` is supported.
- `Numblk` is also used as the `FOR M := 2 TO (Numblk-1)` bound and
  `Totpos-(2*(Numblk-1))`; bounded by grid size, safe to leave INTEGER.

### 5.6 Saturation sentinel rationale
`MaxComplexity = 1_000_000_000` (< MAXLONGINT 2,147,483,647, leaving headroom so
`CombinX+CombinY` in the display total (6079) cannot itself overflow LONGINT:
2e9 < 2.147e9 — **note:** `CombinX+CombinY` could reach 2e9 which is still <
MAXLONGINT, OK; but two saturated rows summed across many rows could exceed it).
**Refinement:** cap `CombinX`/`CombinY` themselves after their row loops too:
```pascal
  IF (CombinX > MaxComplexity) THEN CombinX := MaxComplexity;   (* replaces/augments 4375 *)
  IF (CombinY > MaxComplexity) THEN CombinY := MaxComplexity;
```
so the displayed `CombinX+CombinY` ≤ 2e9 < MAXLONGINT. Choosing 1e9 (not
MAXLONGINT) deliberately leaves that headroom and reads as an obvious "off the
scale" magnitude. The complexity CASE (6085–6098) already routes anything large
to "Does not register on scale (11)", so the exact cap value only affects the
parenthetical total string, not the rating.

### 5.7 Alternatives + why rejected
- **Leave as-is (it's only display):** the displayed numbers are visibly wrong
  (often negative) on large boards — a real, if cosmetic, defect. Cheap to fix
  correctly. But this is the **optional** finding; if descoped, no harm to the
  solver. Recommend fixing given low risk.
- **Compute exact bignum:** absurd over-engineering for a display label.
  Rejected.
- **Float accumulation:** loses the integer "Total of N" semantics and risks
  rounding noise. Rejected.

### 5.8 Blast radius
Confined to `CalComplexity` locals + the two `CombinX/CombinY` post-loop clamps.
No solver path touched. Worst case if mis-implemented: the Info dialog shows a
wrong-but-bounded number — still strictly better than today's wrap.

### 5.9 Test / verification
- Build: `mdtpw` → exit 0, no new warnings (watch for INTEGER/LONGINT mixing
  warnings in the `FOR` bounds — keep loop indices INTEGER, bounds expressions
  evaluate fine).
- **Manual proof:** open Information on a small puzzle (e.g. 5×5) — total matches
  the pre-fix value (no overflow regime). Then on MILLENNI/wide board — pre-fix
  shows a negative/garbage total clamped to 0; post-fix shows a large positive
  saturated total and an "off the scale" rating.
- **Regression:** `solve-all.ps1` — all 51 PASS (solver unaffected; this proves
  no accidental coupling).

### 5.10 Rollback
Revert the decl widenings, the sentinel CONST, and the clamp insertions. The
original `< 0` clamps were never removed, so rollback is clean.

## 6. Finding O — cosmetic `%u` format spec in `CheckGroup`

### 6.1 Defect + location
`NONOGRAM.PAS:2690–2694`:
```pascal
wvsprintf(@T1,
          '%u,%u,%u,%u,%u,%u,%u,%u,%u,%u to '+
          '%u,%u,%u,%u,%u,%u,%u,%u,%u,%u',
          Data);
```
with `Data: ARRAY [1..20] OF INTEGER;` and `T1: ARRAY [0..64] OF CHAR;`
(decls 2639, 2637).

### 6.2 Root cause — GROUND TRUTH verified (source + IR)
The call binds to the **custom runtime** `WVSPrintF`
(`C:\language\mdtpw\runtime_pas\WObjects.pas:4536`), **not** USER32. Verified
two ways:
- **USES order:** NONOGRAM.PAS:65 imports `… WinProcs, WObjects, …`. `WObjects`
  re-declares `WVSPrintF` (4536) after `WinProcs.wvsprintf` (the
  `external 'USER32' name 'wvsprintfA'` at WinProcs.pas:343); the later unit
  wins name resolution. Pascal identifiers are case-insensitive, so the
  lowercase `wvsprintf` call resolves to `WObjects.WVSPrintF`.
- **IR confirmation:** `nonogram.ll:11780` —
  `call i16 @tpw.wobjects.wvsprintf(ptr %v.t1, ptr @.cstr.82, ptr %v.data)`.
  The call targets the WObjects implementation; `wvsprintfA` (declared at
  `nonogram.ll:1958`) is **not** invoked here.

The custom `WVSPrintF` implements only `%d`/`%i`/`%s` (case at WObjects.pas:4661–
4674). `%d`/`%i` (no `l` prefix) call `PutLong(LongInt(ReadIntArg), …)` where
`ReadIntArg` (4556–4560) reads a **16-bit** INTEGER (`PInteger(ArgP)^`) and
advances `ArgP` by **2** bytes — exactly matching `Data`'s 16-bit elements.
**`%u` is unknown** → falls to the `else` branch (4669–4673): it emits the literal
characters `'%'` then `'u'` and **consumes no argument** (`ArgP` not advanced).

**Effect:** the "Possible solution … to …" Recover message renders the literal
format string (`%u,%u,… to %u,…`) instead of the numbers. `Data` is never read,
so **no overread / no crash** — purely cosmetic garbage text.

### 6.3 Fix — before / after
Change every `%u` to `%d`:
```pascal
wvsprintf(@T1,
          '%d,%d,%d,%d,%d,%d,%d,%d,%d,%d to '+
          '%d,%d,%d,%d,%d,%d,%d,%d,%d,%d',
          Data);
```
With `%d`, `WVSPrintF` reads 20 successive 16-bit INTEGERs from `Data` (each
`ReadIntArg` advancing 2 bytes — exactly the layout of
`ARRAY [1..20] OF INTEGER`), formatting the recovered/current clue values.

### 6.4 Buffer-size caveat (must verify, may need T1 enlargement)
`T1` is `ARRAY [0..64] OF CHAR` = 65 bytes incl. NUL. Worst-case output:
20 numbers + 19 commas + `" to "` (4 chars). `Data` holds clue values
(`RecoverData`/`XElements`, bounded by grid ≤100 → 1–3 digits each, all
non-negative). Realistic worst case: 20×3 + 19 + 4 = **83 chars** — this
**exceeds 64** and `WVSPrintF.PutChar` writes unbounded (no length check,
WObjects.pas:4541–4545), so a pathological all-3-digit row would overflow `T1`
on the stack.

**Therefore the fix must also enlarge `T1`** to safely hold the worst case,
e.g.:
```pascal
  T1:      ARRAY [0..127] OF CHAR;   (* was [0..64] *)
```
128 bytes covers 20×3 + 19 + 4 = 83 with headroom. (Note: today's `%u` path
emits the literal format string — ~83 chars too — so this latent overflow
*already exists* in the cosmetic path; `%d` does not create it, but the fix is
the right moment to size `T1` correctly.) Confirm `T1` is used only here in
`CheckGroup` (it is — also at 2742–2744 for the short "Current lowest position:"
string, well within 128).

### 6.5 Alternatives + why rejected
- **Bind to USER32 `wvsprintfA` instead** (via the `WinProcs` name): Win32
  `%d` consumes a **32-bit** vararg, but `Data` is 16-bit INTEGERs → misaligned
  reads / garbage. The runtime author deliberately re-implemented WVSPrintF to
  read 16-bit args for exactly this Win16 corpus (comment WObjects.pas:1121–
  1126). Rejected — would reintroduce a real bug.
- **Reformat into a single `Str()`/`StrCat` loop:** more code, changes the
  message shape, larger blast radius. Rejected — `%d` is the smallest fix.

### 6.6 Blast radius
One format string (40 chars edited `%u`→`%d`) + one decl size bump. Affects only
the Recover "possible solution" diagnostic MessageBox text. No solver impact.

### 6.7 Test / verification
- Build: `mdtpw` → exit 0.
- **Manual proof:** Options | Bad data recovery on a puzzle that reaches the
  "Possible solution found … Display changes?" → Yes branch with a differing
  row. Pre-fix: message shows literal `%u,%u,… to %u,…`. Post-fix: shows the
  actual clue numbers, e.g. `3,1,0,0,… to 2,2,0,0,…`, fitting in `T1`.
- **Regression:** `solve-all.ps1` — all 51 PASS (Recover path not exercised by
  the solve harness; cannot affect solve output).

### 6.8 Rollback
Revert the format string (`%d`→`%u`) and the `T1` size. Independent of all other
findings.
